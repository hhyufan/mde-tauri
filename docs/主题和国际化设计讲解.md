﻿# 主题和国际化设计讲解

## 1. 这篇文档讲什么

这篇文档把“主题设计”和“国际化设计”放在一起讲，因为这两块在当前项目里都属于全局 UI 语言层和视觉层。

重点会回答这几个问题：

- 主题为什么不只是一个 `dark/light` 开关
- 项目如何同时驱动 CSS 变量主题和 Ant Design 主题
- 国际化为什么不只是引入 i18next 就结束了
- Monaco 为什么还要单独做语言桥接
- 主题和语言切换后，界面是如何即时响应的

如果先记一句话，可以把这一块理解成：

`store 中保存主题/语言状态 -> DOM 和 Provider 同步 -> 各组件与第三方库按各自方式消费`

---

## 2. 主题系统不是“写几个 className”，而是三层结构

当前项目的主题系统至少分成 3 层：

1. 主题状态层
   - [../src/store/useThemeStore.js](../src/store/useThemeStore.js#L15-L71)
2. CSS 变量层
   - [../src/assets/styles/index.scss](../src/assets/styles/index.scss#L59-L106)
3. Ant Design token 桥接层
   - [../src/antd/ThemedConfigProvider.jsx](../src/antd/ThemedConfigProvider.jsx#L25-L187)

这说明项目并不是只在 React 组件里判断 `theme === 'dark'`，而是做了真正的全局主题系统。

---

## 3. 主题状态为什么单独放在 `useThemeStore`

[../src/store/useThemeStore.js](../src/store/useThemeStore.js#L15-L71) 是主题状态中心。

它的核心结构如下：

代码位置：[useThemeStore.js:L15-L19](../src/store/useThemeStore.js#L15-L19)
```js
const useThemeStore = create(
  persist(
    (set, get) => ({
      theme: 'light',
      themeUpdatedAt: 0,
```

之所以单独拆一个 store，而不是塞进 `useConfigStore`，是因为主题状态除了“配置值”之外，还会立刻写到 DOM 根节点：

代码位置：[useThemeStore.js:L24-L30](../src/store/useThemeStore.js#L24-L30)
```js
setTheme: (theme, meta = {}) => {
  document.documentElement.dataset.theme = theme;
  set({
    theme,
    themeUpdatedAt: meta.updatedAt ?? Date.now(),
  });
},
```

也就是说，主题切换不是纯 store 行为，而是：

- 改 store
- 改 DOM `data-theme`

这就是为什么它值得单独成一个主题 store。

## 4. CSS 变量主题是整个视觉系统的底座

真正承载视觉主题差异的是 [../src/assets/styles/index.scss](../src/assets/styles/index.scss#L59-L106) 里的 CSS 变量。

亮色主题定义在 `:root`：

代码位置：[index.scss:L59-L84](../src/assets/styles/index.scss#L59-L84)
```scss
:root {
  --bg: #f5f5f7;
  --surface: #fff;
  --sidebar: #fafafa;
  --border: #e5e5e7;
  --text: #2c2c2e;
  --text-sec: #6d6d6f;
  --accent: #4091ff;
  --accent-light: #e8f0ff;
  --success: #34c759;
  --warning: #ff9500;
  --error: #ff3b30;
  --radius: 10px;
  --font-app: 'Poppins', 'Source-HanSan', system-ui, -apple-system, sans-serif;
  --font-editor: 'JetBrains Mono', 'Source-HanSan', monospace;
}
```

暗色主题则挂在 `html[data-theme='dark']`：

代码位置：[index.scss:L86-L106](../src/assets/styles/index.scss#L86-L106)
```scss
html[data-theme='dark'] {
  --bg: rgba(18, 18, 18, 1);
  --surface: rgba(30, 30, 30, 1);
  --sidebar: rgba(24, 24, 24, 1);
  --border: rgb(58, 58, 60);
  --text: rgb(245, 245, 247);
  --text-sec: rgb(152, 152, 157);
  --accent: #5a9fff;
  --accent-light: rgba(90, 159, 255, 0.15);
}
```

这套设计的意义是：

- React 组件不必到处硬编码颜色
- 普通 CSS、组件样式、Ant Design token 都能共用一套设计源
- 主题切换只需要改根节点的 `data-theme`

所以项目的主题系统本质上是“CSS 变量驱动”，而不是“组件条件渲染驱动”。

## 5. 主题初始化为什么要在应用启动时尽早执行

`useThemeStore` 里专门有 `initTheme()`：

代码位置：[useThemeStore.js:L49-L59](../src/store/useThemeStore.js#L49-L59)
```js
initTheme: () => {
  const persisted = localStorage.getItem('mde-theme');
  const savedTheme = persisted ? JSON.parse(persisted)?.state?.theme : null;
  const initial = savedTheme || (window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light');
  document.documentElement.dataset.theme = initial;
  set((state) => ({
    theme: initial,
    themeUpdatedAt: state.themeUpdatedAt || 0,
  }));
},
```

它的优先级是：

1. 先读本地持久化主题
2. 没有时回退系统主题偏好

而且 store 回填时还会立即把主题应用到 DOM，避免首屏闪烁：

代码位置：[useThemeStore.js:L63-L68](../src/store/useThemeStore.js#L63-L68)
```js
onRehydrateStorage: () => (state) => {
  if (state?.theme) {
    document.documentElement.dataset.theme = state.theme;
  }
},
```

这说明项目在主题系统上关注的不只是“切换成功”，还关注“首屏别闪”。

## 6. 为什么还要做 `ThemedConfigProvider`

如果项目只有自定义 CSS，那么到这里其实已经够了。  
但当前项目大量使用了 Ant Design 组件，所以还需要把自定义主题同步到 antd token 系统。

这个桥接层就是 [../src/antd/ThemedConfigProvider.jsx](../src/antd/ThemedConfigProvider.jsx#L69-L187)。

### 6.1 antd token 不是手写一套，而是从 CSS 变量反推

核心方法是 `buildTokens()`：

代码位置：[ThemedConfigProvider.jsx:L69-L92](../src/antd/ThemedConfigProvider.jsx#L69-L92)
```js
function buildTokens() {
  return {
    colorPrimary: readCssVar('--accent', '#4091ff'),
    colorInfo: readCssVar('--info', '#4091ff'),
    colorSuccess: readCssVar('--success', '#34c759'),
    colorWarning: readCssVar('--warning', '#ff9500'),
    colorError: readCssVar('--error', '#ff3b30'),
    colorText: readCssVar('--text', '#2c2c2e'),
    colorTextSecondary: readCssVar('--text-sec', '#6d6d6f'),
    colorBgBase: readCssVar('--bg', '#f5f5f7'),
    colorBgContainer: readCssVar('--surface', '#ffffff'),
    borderRadius: parseRadius(readCssVar('--radius-sm', '6px'), 6),
    borderRadiusLG: parseRadius(readCssVar('--radius', '10px'), 10),
    fontFamily:
      readCssVar('--font-app', '')
      || '"Poppins", system-ui, -apple-system, "Segoe UI", sans-serif',
  };
}
```

这套设计很聪明，因为它保证：

- CSS 层和 antd 层不会维护两套独立主题源
- 改一套 CSS 变量，就能带动 antd token 更新

### 6.2 主题切换后为什么要下一帧重新读 token

代码位置：[ThemedConfigProvider.jsx:L107-L112](../src/antd/ThemedConfigProvider.jsx#L107-L112)
```js
useEffect(() => {
  const raf = requestAnimationFrame(() => {
    setTokenVersion((v) => v + 1);
  });
  return () => cancelAnimationFrame(raf);
}, [theme]);
```

这里的意思是：

- `theme` 改变后，DOM 上的 CSS 变量也会变化
- 但这些变化不是同一拍就能被 `getComputedStyle` 读到
- 所以要下一帧再重新取 token

这就是一个典型的“CSS 变量主题系统和组件库主题系统之间的时序协调”问题。

## 8. 国际化设计不是只引 i18next 就结束了

国际化入口在 [../src/i18n/index.js](../src/i18n/index.js#L20-L72)。

文件开头就写清楚了它的职责：

代码位置：[i18n/index.js:L1-L6](../src/i18n/index.js#L1-L6)
```js
/**
 * 应用国际化初始化入口。
 *
 * 本文件负责确定初始语言、注册中英文资源、初始化 i18next，并在语言切换时
 * 同步更新根节点的 `lang` 属性，供界面与可访问性能力复用。
 */
```

所以 i18n 在这个项目里承担的是：

- 决定初始语言
- 注册语言资源
- 初始化 i18next
- 维护 `<html lang>`

这已经不是简单的 `useTranslation` 使用层了。

## 9. 初始语言为什么要自己读，而不是完全依赖 i18next-browser-languagedetector

项目没有完全交给探测器，而是自己写了 `readInitialLang()`：

代码位置：[i18n/index.js:L20-L32](../src/i18n/index.js#L20-L32)
```js
function readInitialLang() {
  try {
    const persisted = localStorage.getItem('i18nextLng')
      || JSON.parse(localStorage.getItem('mde-config') || 'null')?.state?.language;
    if (persisted) {
      return /^zh/i.test(persisted) ? 'zh' : 'en';
    }
  } catch {}
  if (typeof navigator !== 'undefined' && /^zh/i.test(navigator.language || '')) {
    return 'zh';
  }
  return 'en';
}
```

优先级是：

1. `i18nextLng`
2. `mde-config.language`
3. `navigator.language`

这说明国际化系统考虑了“历史偏好”和“应用配置”两个来源，而不只是浏览器默认语言。

## 10. 为什么语言资源使用扁平 key

i18n 初始化时显式关闭了 key 分隔符：

代码位置：[i18n/index.js:L37-L64](../src/i18n/index.js#L37-L64)
```js
i18n.init({
  resources: {
    en: { translation: enUS },
    zh: { translation: zhCN },
  },
  lng: initialLang,
  fallbackLng: 'en',
  keySeparator: false,
  nsSeparator: false,
})
```

这说明项目里的语言 JSON 使用的是扁平 key，例如：

- `toolbar.bold`
- `settings.general.language`

而不是层层嵌套对象。

这样设计的好处是：

- key 在代码里更稳定
- Vite 开发环境和生产构建行为更一致
- 不会因为 `.` 被 i18next 当成路径拆分出奇怪问题

## 11. 为什么 `<html lang>` 也要跟着变

`i18n` 初始化完成后，项目会维护根节点语言：

代码位置：[i18n/index.js:L34-L35](../src/i18n/index.js#L34-L35) + [i18n/index.js:L69-L72](../src/i18n/index.js#L69-L72)（组合摘录）
```js
const initialLang = readInitialLang();
document.documentElement.lang = initialLang;

i18n.on('languageChanged', (lng) => {
  const key = /^zh/i.test(lng) ? 'zh' : 'en';
  document.documentElement.lang = key;
});
```

这意味着项目考虑的不只是界面文字翻译，还有：

- 浏览器语言上下文
- 屏幕阅读器和辅助技术
- 部分控件/输入法对页面语言的感知

所以国际化设计不是“组件渲染语言包”这么简单。

## 12. 为什么 antd 的国际化还要单独做一层

i18next 只负责项目自己的业务文案。  
Ant Design 组件内部文案，例如：

- 日期选择器
- 分页器
- Modal 内置按钮文案

并不会自动从 i18next 里拿。

所以 [../src/antd/ThemedConfigProvider.jsx](../src/antd/ThemedConfigProvider.jsx#L100-L187) 还要负责 antd locale：

代码位置：[ThemedConfigProvider.jsx:L102-L105](../src/antd/ThemedConfigProvider.jsx#L102-L105) + [ThemedConfigProvider.jsx:L134-L142](../src/antd/ThemedConfigProvider.jsx#L134-L142)（组合摘录）
```js
const language = useConfigStore((s) => s.language);
const locale = language === 'zh' && zhLoaded && zhCnLocale ? zhCnLocale : enUS;

return (
  <ConfigProvider
    locale={locale}
    theme={{ ... }}
  >
```

而且中文语言包是懒加载的：

代码位置：[ThemedConfigProvider.jsx:L25-L34](../src/antd/ThemedConfigProvider.jsx#L25-L34)
```js
function loadZhCn() {
  if (zhCnLocale) return null;
  if (!zhCnPromise) {
    zhCnPromise = import('antd/locale/zh_CN').then((mod) => {
      zhCnLocale = mod.default || mod;
      return zhCnLocale;
    });
  }
  return zhCnPromise;
}
```

这说明项目在国际化设计上不只是“正确”，还考虑了首包体积。

## 14. 主题和国际化为什么都要和设置模块联动

SettingsModal 中的 `handleChange()` 就体现了这种联动：

代码位置：[SettingsModal.jsx:L76-L81](../src/components/overlays/SettingsModal.jsx#L76-L81)
```js
function handleChange(key, value) {
  config.setConfig(key, value);
  if (key === 'language') {
    i18n.changeLanguage(value === 'zh' ? 'zh' : 'en');
  }
}
```

主题则通过 `setTheme()` 直接落到 `useThemeStore`。

也就是说：

- 设置模块负责提供用户入口
- 主题系统负责更新视觉变量和 antd token
- 国际化系统负责更新业务文案、组件文案和根节点语言

所以三者是“入口-实现-消费”的关系，而不是彼此替代。

## 16. 结论

当前项目的主题和国际化设计，真正厉害的地方不是“能切换”，而是它们都做成了完整的全局系统：

- 主题系统
  - `useThemeStore` 维护状态
  - `data-theme` 驱动 CSS 变量
  - `ThemedConfigProvider` 把 CSS 变量桥接到 antd token
- 国际化系统
  - `i18n/index.js` 管理资源、检测、初始化和 `<html lang>`
  - `ThemedConfigProvider` 同步 antd locale
  - `monacoLocale` 单独处理 Monaco 运行时和首次加载时的文案

所以从架构上看，这两块都不是“设置页上的一个选项”，而是贯穿全局渲染层的基础设施。
