﻿# 配置导入导出实现讲解

## 1. 这篇文档讲什么

这篇文档专门讲当前项目里的“配置导入导出”实现。  
这里既包括：

- 本地 JSON 导出
- 本地 JSON 导入

也包括它背后的设置快照设计：

- 哪些设置会被导出
- 快照格式为什么这样设计
- 导入后如何回放到多个 store
- 这套快照结构为什么还能直接复用到云端配置同步

如果只先记一句话，可以把它理解成：

`多个 store -> 统一设置快照 -> 导出为 JSON / 导入并回放 -> 与云端同步共用同一套结构`

---

## 2. 配置导入导出不是直接把 store 原样 dump 出去

项目没有简单地把 `useConfigStore` 或 `localStorage` 原样导出，而是专门写了一套设置快照工具，在 [../src/utils/settingsSync.js](../src/utils/settingsSync.js#L8-L111)。

文件开头就说明了它的定位：

代码位置：[settingsSync.js:L8-L19](../src/utils/settingsSync.js#L8-L19)
```js
/**
 * 配置快照与回放工具。
 *
 * 负责把多个配置相关 store 统一抽取为稳定结构，并支持导出、导入和云端同步共用。
 */
```

这说明项目对配置导入导出的理解不是“持久化复制”，而是“构造一个稳定、可迁移、可同步的设置快照格式”。

---

## 3. 为什么要先构造“本地设置快照”

核心函数是 `getLocalSettingsSnapshot()`：

代码位置：[settingsSync.js:L21-L55](../src/utils/settingsSync.js#L21-L55)
```js
export function getLocalSettingsSnapshot() {
  const config = useConfigStore.getState();
  const theme = useThemeStore.getState();
  const editor = useEditorStore.getState();
  const updatedAt = Math.max(
    Number(config.configUpdatedAt || 0),
    Number(theme.themeUpdatedAt || 0),
    Number(editor.uiStateUpdatedAt || 0),
  );

  return {
    theme: theme.theme,
    language: config.language,
    fontSize: config.fontSize,
    previewFontSize: config.previewFontSize,
    fontFamily: config.fontFamily,
    lineHeight: config.lineHeight,
    previewLineHeight: config.previewLineHeight,
    previewZoomSync: config.previewZoomSync,
    tabSize: config.tabSize,
    wordWrap: config.wordWrap,
    lineNumbers: config.lineNumbers,
    minimap: sanitizeMinimap(config.minimap),
    autoSave: config.autoSave,
    workspacePath: config.workspacePath,
    editorState: {
      sidebarVisible: editor.sidebarVisible,
      sidebarView: editor.sidebarView,
      viewMode: editor.viewMode,
      toolbarVisible: editor.toolbarVisible,
    },
    protocolVersion: SYNC_PROTOCOL_VERSION,
    updatedAt,
  };
}
```

这段代码说明，配置导出并不是只包含“设置页上的表单值”，而是聚合了至少三块状态：

- `useConfigStore`
  - 通用配置
- `useThemeStore`
  - 主题
- `useEditorStore`
  - 一部分可同步的编辑器 UI 状态

所以项目里的配置导入导出，本质上是“多 store 快照化”。

## 4. 为什么快照里要有 `updatedAt` 和 `protocolVersion`

这两个字段很关键。

### 4.1 `updatedAt`

代码位置：[settingsSync.js:L25-L29](../src/utils/settingsSync.js#L25-L29)
```js
const updatedAt = Math.max(
  Number(config.configUpdatedAt || 0),
  Number(theme.themeUpdatedAt || 0),
  Number(editor.uiStateUpdatedAt || 0),
);
```

它代表的是：

- 这份设置快照最后一次有效变更时间

这样做的意义是：

- 本地导入时可以保留时间语义
- 云端同步时可以比较新旧
- 多来源设置回放时不会完全丢失时间信息

### 4.2 `protocolVersion`

代码位置：[settingsSync.js:L52-L53](../src/utils/settingsSync.js#L52-L53)
```js
protocolVersion: SYNC_PROTOCOL_VERSION,
```

这个字段说明项目从一开始就把设置快照看成“协议对象”，而不是临时 JSON。  
它为以后格式升级、兼容策略预留了空间。

## 5. 为什么 `minimap` 要专门 sanitize

在快照构造时，项目没有直接把 minimap 配置整个塞进去，而是先走 `sanitizeMinimap()`：

代码位置：[settingsSync.js:L11-L13](../src/utils/settingsSync.js#L11-L13)
```js
function sanitizeMinimap(minimap) {
  return { enabled: !!minimap?.enabled };
}
```

这说明配置导入导出追求的是：

- 保留真正需要同步的字段
- 去掉运行时噪声
- 保持快照结构稳定

所以它不是“深拷贝当前 store”，而是“提纯成稳定协议结构”。

---

## 6. 导出文件为什么还要再包一层载荷

真正用于导出到文件的不是裸 `settings`，而是 `buildSettingsExportPayload()`：

代码位置：[settingsSync.js:L90-L97](../src/utils/settingsSync.js#L90-L97)
```js
export function buildSettingsExportPayload() {
  return {
    type: 'mde-settings',
    version: 1,
    exportedAt: new Date().toISOString(),
    settings: getLocalSettingsSnapshot(),
  };
}
```

这层包装有几个好处：

- `type`
  - 明确表明这是什么文件
- `version`
  - 方便未来调整格式
- `exportedAt`
  - 便于定位导出时间
- `settings`
  - 真正的设置内容本体

也就是说，项目导出的不是“设置对象”，而是“带元数据的设置包”。

## 7. 本地导出流程是怎么走的

导出入口在 [../src/components/overlays/SettingsModal.jsx](../src/components/overlays/SettingsModal.jsx#L127-L168) 的 `handleExportJson()`：

代码位置：[SettingsModal.jsx:L127-L143](../src/components/overlays/SettingsModal.jsx#L127-L143)
```js
async function handleExportJson() {
  setCloudBusy(true);
  try {
    const path = await save({
      defaultPath: 'mde-settings.json',
      filters: [{ name: 'JSON', extensions: ['json'] }],
    });
    if (!path) return;
    const payload = buildSettingsExportPayload();
    await writeTextFile(path, JSON.stringify(payload, null, 2));
    notify('success', t('settings.cloud.exportJson'), t('settings.cloud.exportSuccess'));
  } catch (err) {
    notify('error', t('notification.error'), err?.message || String(err));
  } finally {
    setCloudBusy(false);
  }
}
```

这条链路可以拆成：

1. 弹出保存对话框
2. 构造标准设置包
3. 格式化为 JSON
4. 写入本地文件
5. 给出成功/失败通知

所以导出流程的本质是：

`SettingsModal -> buildSettingsExportPayload -> JSON -> writeTextFile`

## 8. 导入为什么要支持两种格式

导入解析函数是 `parseSettingsImportPayload()`：

代码位置：[settingsSync.js:L102-L111](../src/utils/settingsSync.js#L102-L111)
```js
export function parseSettingsImportPayload(rawText) {
  const parsed = JSON.parse(rawText);
  const settings = parsed?.settings && typeof parsed.settings === 'object'
    ? parsed.settings
    : parsed;
  if (!settings || typeof settings !== 'object') {
    throw new Error('Invalid settings payload');
  }
  return settings;
}
```

这里体现了一个很实用的兼容策略：

- 如果导入的是完整导出包
  - 取 `parsed.settings`
- 如果导入的是纯 settings 对象
  - 也接受

这意味着项目对导入格式没有强制得特别死，而是尽量提高可恢复性和兼容性。

## 9. 导入后为什么不是简单覆盖一个 store

真正应用外部快照的是 `applySettingsSnapshot()`：

代码位置：[settingsSync.js:L62-L85](../src/utils/settingsSync.js#L62-L85)
```js
export function applySettingsSnapshot(snapshot = {}) {
  const updatedAt = Number(snapshot.updatedAt || Date.now());
  useThemeStore.getState().setTheme(snapshot.theme || 'light', { updatedAt });
  const language = snapshot.language ?? 'en';
  const fontSize = snapshot.fontSize ?? 14;
  const lineHeight = snapshot.lineHeight ?? 24;
  useConfigStore.getState().loadConfig({
    language,
    fontSize,
    previewFontSize: snapshot.previewFontSize ?? fontSize,
    fontFamily: snapshot.fontFamily || 'JetBrains Mono',
    lineHeight,
    previewLineHeight: snapshot.previewLineHeight ?? lineHeight,
    previewZoomSync: snapshot.previewZoomSync ?? true,
    tabSize: snapshot.tabSize ?? 2,
    wordWrap: snapshot.wordWrap ?? true,
    lineNumbers: snapshot.lineNumbers ?? true,
    minimap: sanitizeMinimap(snapshot.minimap),
    autoSave: snapshot.autoSave ?? true,
    workspacePath: snapshot.workspacePath ?? '',
  }, { updatedAt });
  i18n.changeLanguage(language === 'zh' ? 'zh' : 'en');
  useEditorStore.getState().applySyncedUiState(snapshot.editorState || {}, { updatedAt });
}
```

这段代码说明，导入并不是“整份 JSON 覆盖一个 store”，而是：

- 主题写入 `useThemeStore`
- 通用配置写入 `useConfigStore`
- 语言同步更新 i18n
- 编辑器 UI 状态回放到 `useEditorStore`

也就是说，导入本质上是一次“多 store 回放”。

## 10. 本地导入流程是怎么走的

SettingsModal 中的入口是 `handleImportJson()`：

代码位置：[SettingsModal.jsx:L150-L168](../src/components/overlays/SettingsModal.jsx#L150-L168)
```js
async function handleImportJson() {
  setCloudBusy(true);
  try {
    const path = await open({
      multiple: false,
      filters: [{ name: 'JSON', extensions: ['json'] }],
    });
    if (!path || Array.isArray(path)) return;
    const raw = await readTextFile(path);
    const settings = parseSettingsImportPayload(raw);
    applySettingsSnapshot(settings);
    i18n.changeLanguage((settings.language || 'en') === 'zh' ? 'zh' : 'en');
    notify('success', t('settings.cloud.importJson'), t('settings.cloud.importSuccess'));
  } catch (err) {
    notify('error', t('notification.error'), err?.message || String(err));
  } finally {
    setCloudBusy(false);
  }
}
```

这条链路是：

1. 选择 JSON 文件
2. 读取文本
3. 解析导入包或纯 settings 对象
4. 把快照回放到多个 store
5. 主动同步语言
6. 给出结果通知

其中第 4 步和第 5 步最关键，因为这才真正让导入后的状态立即生效。

## 11. 为什么配置导入导出和云同步能共用同一套结构

这是当前设计很漂亮的一点。

你会发现：

- 本地导出用 `buildSettingsExportPayload()`
- 本地导入用 `parseSettingsImportPayload() + applySettingsSnapshot()`
- 云端同步也依赖 `getLocalSettingsSnapshot()` / `applySettingsSnapshot()`

也就是说，项目不是分别做了：

- 一套“本地导入导出格式”
- 一套“云同步格式”

而是：

- 先定义统一设置快照
- 再让本地 JSON 和远端配置同步都复用它

这让设计更稳定，也减少了两套格式逐渐偏离的风险。

## 12. 配置导入导出和设置模块、主题、国际化是什么关系

从职责上看：

- 设置模块
  - 负责提供导入导出入口按钮
- 配置快照工具
  - 负责组装、解析、回放
- 主题系统
  - 负责响应导入后的主题变化
- 国际化系统
  - 负责响应导入后的语言变化

所以配置导入导出并不是一个孤立功能，而是“设置基础设施”的一部分。

## 14. 结论

当前项目的配置导入导出，真正的核心不是“存成 JSON”，而是“把多个配置相关 store 统一提炼成稳定快照，再支持回放”：

- `getLocalSettingsSnapshot()`
  - 负责从多个 store 抽取设置真值
- `buildSettingsExportPayload()`
  - 负责构造可落地的导出包
- `parseSettingsImportPayload()`
  - 负责兼容解析不同导入格式
- `applySettingsSnapshot()`
  - 负责把导入结果回放到主题、配置、语言和编辑器 UI 状态

所以从架构上看，配置导入导出不是附属功能，而是一套能被本地 JSON 和云同步同时复用的设置快照协议。
