# 内容检索和文件检索怎么实现的

## 1. 这篇文档讲什么

这篇文档解释当前项目里的搜索模块是怎么实现的，重点区分两类能力：

- 文件检索：按文件名搜索
- 内容检索：按文件内容搜索

两者共用一套前端搜索弹窗和桥接 API，但在 Rust 侧会分成两条不同的执行路径。

---

## 2. 搜索模块的整体结构

搜索能力整体可以分成 4 层：

- 入口层：快捷键和标题栏搜索入口
- 前端编排层：`SearchModal` 负责输入、模式切换、结果展示和打开文件
- 桥接层：`tauriApi.searchFiles()` 负责分流到桌面端或 Android SAF
- 原生搜索层：Rust `search_files` 负责真正的目录遍历和内容扫描

- 全局快捷键入口：[App.jsx:L182-L225](../src/App.jsx#L182-L225)
- 标题栏搜索入口：[TitleBar.jsx:L30-L63](../src/components/layout/title-bar/TitleBar.jsx#L30-L63)
- 搜索弹窗总入口：[SearchModal.jsx:L29-L257](../src/components/overlays/SearchModal.jsx#L29-L257)

## 5. 搜索请求是怎么发起的

搜索并不是每按一个键就立即发请求，而是先经过防抖。

代码位置：[SearchModal.jsx:L60-L86](../src/components/overlays/SearchModal.jsx#L60-L86)

```js
const doSearch = useCallback(
  debounce(async (q, dir, isContent) => {
    if (!q.trim() || !dir) {
      setResults([]);
      setLoading(false);
      return;
    }
    try {
      const res = await searchFiles(dir, q.trim(), isContent, 80);
      setResults(res);
      setSelectedIndex(0);
    } catch (_) {
      setResults([]);
    }
    setLoading(false);
  }, 250),
  [],
);

useEffect(() => {
  if (!query.trim()) {
    setResults([]);
    return;
  }
  setLoading(true);
  doSearch(query, currentDir, searchContent);
}, [query, searchContent, currentDir, doSearch]);
```

这里能看出几个关键点：

- 防抖时间是 `250ms`
- 搜索根目录来自当前工作目录 `currentDir`
- 是否做内容搜索由 `searchContent` 决定
- 最多返回 `80` 条结果给搜索弹窗

也就是说，前端的职责主要是“组织请求参数”，真正搜索并不在前端完成。

## 8. 前端桥接层 `searchFiles()` 怎么分流

前端统一搜索 API 在 [tauriApi.js:L292-L319](../src/utils/tauriApi.js#L292-L319)。

代码位置：[tauriApi.js:L292-L319](../src/utils/tauriApi.js#L292-L319)

```js
export async function searchFiles(dirPath, query, searchContent = false, maxResults = 100) {
  if (isSafUri(dirPath)) {
    try {
      const q = (query || '').toLowerCase();
      if (!q) return [];
      const children = await safListFolder(dirPath);
      const matches = [];
      for (const c of children) {
        if (matches.length >= maxResults) break;
        if (c.name.toLowerCase().includes(q)) {
          matches.push({
            name: c.name,
            path: c.path,
            is_dir: !!c.is_dir,
            matched_line: null,
            line_number: null,
          });
        }
      }
      return matches;
    } catch (_) {
      return [];
    }
  }
  return invoke('search_files', { dirPath, query, searchContent, maxResults });
}
```

这段代码非常关键，因为它说明：

- 桌面端会调用 Rust `search_files`
- Android SAF 场景下降级成“当前目录单层文件名匹配”
- SAF 不支持深层内容扫描，这是出于性能考虑

所以如果你发现 Android 上的搜索能力没有桌面端强，不是 bug，而是桥接层有意为之。

## 10. 文件检索是怎么实现的

文件检索走的是目录递归遍历分支。

关键代码位置：

- 总调度的文件名分支：[lib.rs:L832-L844](../src-tauri/src/lib.rs#L832-L844)
- 文件名递归遍历核心：[lib.rs:L729-L769](../src-tauri/src/lib.rs#L729-L769)

其核心思想是：

- 递归遍历目录树
- 只比较文件名是否包含查询串
- 匹配后构造 `SearchResult`
- 达到上限后提前停止

这类搜索的成本主要来自文件系统目录遍历，而不是文件内容读取，所以相对更轻量。

## 11. 内容检索是怎么实现的

内容检索比文件检索复杂得多，它需要真正读文件内容。

关键代码位置：

- 收集 Markdown 文件路径：[lib.rs:L675-L694](../src-tauri/src/lib.rs#L675-L694)
- 按行扫描内容：[lib.rs:L696-L727](../src-tauri/src/lib.rs#L696-L727)
- 总调度中的内容搜索分支：[lib.rs:L790-L831](../src-tauri/src/lib.rs#L790-L831)

Rust 侧的设计分成两步：

1. 先递归收集候选 Markdown 文件路径
2. 再并发读取文件内容并逐行匹配

从 [lib.rs:L790-L831](../src-tauri/src/lib.rs#L790-L831) 可以看出，它使用了 `spawn_blocking` 把同步文件系统操作放到线程池里，避免阻塞异步运行时。

代码位置：[lib.rs:L790-L845](../src-tauri/src/lib.rs#L790-L845)

```rust
if search_content {
    let root_clone = root_path.clone();
    let md_paths = async_runtime::spawn_blocking(move || {
        let mut paths = Vec::new();
        collect_md_paths(&root_clone, &mut paths);
        paths
    })
    .await
    .map_err(|e| e.to_string())?;

    let handles: Vec<_> = md_paths
        .into_iter()
        .map(|path| {
            let q = Arc::clone(&query_lower);
            async_runtime::spawn_blocking(move || match fs::read_to_string(&path) {
                Ok(content) => search_content_lines(&path, &content, &q),
                Err(_) => vec![],
            })
        })
        .collect();
    // ...
}
```

这个实现说明内容检索并不是“边遍历目录边同步读文件”的朴素做法，而是做了一层线程池并发优化。

## 12. 搜索结果结构为什么能同时支持两种模式

Rust 侧统一返回 `SearchResult`，这就是前端可以用同一个结果列表同时显示“文件名搜索结果”和“内容搜索结果”的原因。

相关代码位置：

- 结果结构体：[lib.rs:L651-L662](../src-tauri/src/lib.rs#L651-L662)
- 搜索结果渲染：[SearchModal.jsx:L214-L245](../src/components/overlays/SearchModal.jsx#L214-L245)

对于文件检索：

- `matched_line` 和 `line_number` 往往为空

对于内容检索：

- `matched_line` 保存命中的那一行文本
- `line_number` 保存命中行号

这样前端就能根据结果字段决定是否展示命中行预览，以及是否在打开文件后跳行。

## 13. 这套搜索实现的一个重要限制

桥接层已经明确写了：Android SAF 场景下降级为单层文件名搜索。

这意味着：

- 桌面端支持递归文件名搜索
- 桌面端支持内容检索
- Android SAF 只做当前目录单层名称匹配

相关代码位置：[tauriApi.js:L293-L317](../src/utils/tauriApi.js#L293-L317)

这是跨平台权衡，不是简单缺功能。原因在于 SAF 深层遍历和内容扫描代价很高，用户体验会变差。

## 15. 结论

当前项目的搜索并不是单纯的一个“搜索框”，而是一套分层实现：

- 前端负责交互和结果展示
- `tauriApi` 负责跨平台桥接
- Rust 负责桌面端真正的目录遍历和文本扫描

文件检索和内容检索共用 UI 和桥接入口，但在 Rust 侧会分裂为两条不同的执行路径。只要把 `SearchModal -> tauriApi.searchFiles -> Rust search_files` 这条链路吃透，整个搜索模块就能看明白。
