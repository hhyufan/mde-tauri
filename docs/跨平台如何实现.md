# 跨平台如何实现

## 1. 这篇文档讲什么

这篇文档专门解释当前项目在前端、Tauri、Android 之间是如何做跨平台适配的。
重点不是泛泛地说“用了 Tauri”，而是讲清楚下面这条真实链路：

`业务调用 -> JS bridge -> 桌面端 Rust command / Android SAF bridge -> 原生文件系统或系统能力`

当前项目的跨平台实现，主要围绕两类运行环境展开：

- 桌面端
  - Windows / macOS / Linux，走 Tauri `invoke` 到 Rust command
- Android 端
  - 走 `content://` SAF URI 和 Kotlin bridge

项目并没有在每个业务点手写一堆 `if Android else Desktop`，而是通过一层统一桥接把平台差异往下压。

---

## 2. 跨平台总结构：不是组件直接碰原生，而是统一走桥接层

整个跨平台适配的最核心入口是 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L1-L30)。

这个文件开头就直接说明了自己的定位：

代码位置：[tauriApi.js:L1-L6](../src/utils/tauriApi.js#L1-L6)
```js
/**
 * Tauri 与 Android SAF 的统一文件能力适配层。
 *
 * 本文件对上层暴露稳定的“按路径工作”接口，并在内部根据普通文件路径与
 * `content://` SAF URI 自动分流到 Rust 命令或 Android Bridge，屏蔽平台差异。
 */
```

也就是说，上层业务只需要继续用“路径式 API”思维工作，例如：

- `readFileContent(path)`
- `saveFile(path, content, encoding)`
- `getDirectoryContents(dirPath)`
- `renameFile(oldPath, newPath)`

但 `tauriApi` 会自己判断：

- 这是普通路径
  - 走 Rust `invoke`
- 这是 `content://` URI
  - 走 Android SAF bridge

这就是整个跨平台架构最重要的抽象层。

## 4. 真正的跨平台核心：`tauriApi` 如何把一套 API 分流到两条原生链路

### 4.1 路径类型是分流依据

在 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L103-L126) 中，文件读取就是一个典型例子：

代码位置：[tauriApi.js:L103-L126](../src/utils/tauriApi.js#L103-L126)
```js
export async function readFileContent(path) {
  if (isSafUri(path)) {
    try {
      const { content, encoding, lineEnding, size } = await safReadFileText(path);
      const fileName = safDisplayName(path);
      return {
        success: true,
        message: 'File read successfully',
        content,
        file_path: path,
        file_name: fileName,
        encoding,
        line_ending: lineEnding,
        size,
      };
    } catch (e) {
      return {
        success: false,
        message: `Failed to read file: ${e?.message || e}`,
      };
    }
  }
  return invoke('read_file_content', { path });
}
```

这段代码已经把项目的跨平台策略讲得很清楚：

- 上层 API 不变
- 如果是 `content://`
  - 走 Android SAF
- 否则
  - 走 Tauri Rust command

保存、列目录、重命名、删除这些操作也是一样的分流模式。

### 4.2 统一 API 让业务层保持“按路径思维”

比如保存文件：

代码位置：[tauriApi.js:L142-L163](../src/utils/tauriApi.js#L142-L163)
```js
export async function saveFile(filePath, content, encoding) {
  if (isSafUri(filePath)) {
    const result = await safWriteFileText(filePath, content);
    return {
      success: true,
      message: 'File saved successfully',
      file_path: filePath,
      file_name: safDisplayName(filePath),
      encoding: encoding || 'UTF-8',
      line_ending: detectLineEnding(content),
      size: result?.size,
    };
  }
  return invoke('save_file', { filePath, content, encoding });
}
```

列目录也是：

代码位置：[tauriApi.js:L197-L202](../src/utils/tauriApi.js#L197-L202)
```js
export async function getDirectoryContents(dirPath) {
  if (isSafUri(dirPath)) {
    return safListFolder(dirPath);
  }
  return invoke('get_directory_contents', { dirPath });
}
```

所以跨平台的关键不是“每个平台做完全不同的业务逻辑”，而是“用同一套上层接口，内部做平台路由”。

## 5. Android 为什么要走 SAF，而不是直接让 Rust 读写文件

这是理解项目跨平台实现的关键。

Android 上用户从系统文件选择器拿到的通常不是传统磁盘路径，而是 `content://` URI。
这种 URI 背后是 `DocumentsProvider`，不是 Rust 可以直接 `std::fs::read` 的真实路径。

所以项目单独实现了一条 Android 文件桥接链：

- JS 层
  - [../src/utils/androidSaf.js](../src/utils/androidSaf.js#L1-L13)
- Activity 层
  - [../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt](../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt#L98-L102)
- 原生 SAF 操作层
  - [../src-tauri/gen/android/app/src/main/java/com/mde/app/SafBridge.kt](../src-tauri/gen/android/app/src/main/java/com/mde/app/SafBridge.kt#L18-L29)

`androidSaf.js` 文件开头就把这个协议写得很清楚：

代码位置：[androidSaf.js:L1-L8](../src/utils/androidSaf.js#L1-L8)
```js
/**
 * Android SAF（Storage Access Framework）桥接模块。
 *
 * Kotlin 侧（`SafBridge` + `MainActivity`）通过 `window.AndroidSaf`
 * 暴露同步 JavaScript 接口。文件/目录选择器本质上是异步的，因为它会拉起外部 Activity，
 * 所以原生桥先返回一个数字回调 id，稍后再在主线程调用
 * `window.__androidSafResolve(id, uri)` 回传结果。
 */
```

这就是为什么 Android 侧没有复用桌面端那套 Rust 文件 API。

## 6. Android SAF 桥接是怎么做成 Promise 风格的

Android 选择器本质是“先拉起外部 Activity，再异步返回结果”，而 JS 业务层又希望是 `await pickFile()` 这种 Promise 形式。
所以项目在 [../src/utils/androidSaf.js](../src/utils/androidSaf.js#L19-L31) 里做了一个 callback id 到 Promise 的包装。

核心逻辑如下：

代码位置：[androidSaf.js:L19-L31](../src/utils/androidSaf.js#L19-L31) + [androidSaf.js:L69-L77](../src/utils/androidSaf.js#L69-L77)（组合摘录）
```js
const pending =
  (typeof window !== 'undefined' && window.__androidSafPending) ||
  new Map();

if (typeof window !== 'undefined') {
  window.__androidSafPending = pending;
  if (!window.__androidSafResolve) {
    window.__androidSafResolve = (id, uri) => {
      const entry = pending.get(id);
      if (!entry) return;
      pending.delete(id);
      entry.resolve(uri || null);
    };
  }
}

function awaitCallback(id) {
  return new Promise((resolve, reject) => {
    if (!id || id <= 0) {
      reject(new Error('Invalid SAF callback id'));
      return;
    }
    pending.set(id, { resolve, reject });
  });
}
```

真正的 `pickFile()` 就变得很自然：

代码位置：[androidSaf.js:L102-L107](../src/utils/androidSaf.js#L102-L107)
```js
export async function pickFile(mimeTypes = ['*/*']) {
  const b = getBridge();
  if (!b) throw new Error('AndroidSaf bridge unavailable');
  const id = b.pickFile(JSON.stringify(mimeTypes));
  return awaitCallback(id);
}
```

这说明 Android 桥接层的工作不是只“转发原生能力”，而是把原生异步选择器协议转成更适合前端消费的 Promise 协议。

## 7. `MainActivity` 在跨平台链路里扮演什么角色

[../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt](../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt#L98-L102) 是 Android 侧的“桥头堡”。

### 7.1 它负责把原生桥对象注入到 WebView

代码位置：[MainActivity.kt:L98-L102](../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt#L98-L102)
```kt
override fun onWebViewCreate(webView: WebView) {
    super.onWebViewCreate(webView)
    webViewRef = webView
    webView.addJavascriptInterface(safBridge, "AndroidSaf")
}
```

所以前端的 `window.AndroidSaf` 不是凭空出现的，而是这里注入的。

### 7.2 它负责注册系统选择器并保存回调 ID

代码位置：[MainActivity.kt:L15-L17](../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt#L15-L17) + [MainActivity.kt:L44-L53](../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt#L44-L53)（组合摘录）
```kt
private var pendingFolderCb: Long = 0L
private var pendingFileCb: Long = 0L
private var pendingCreateCb: Long = 0L

folderLauncher = registerForActivityResult(
    ActivityResultContracts.OpenDocumentTree()
) { uri ->
    val cb = pendingFolderCb
    pendingFolderCb = 0L
    if (uri != null) {
        takePersistableSafe(uri)
    }
    resolveCallback(cb, uri?.toString())
}
```

这说明 `MainActivity` 负责：

- 注册 Android 系统选择器
- 暂存本次回调 id
- 在结果返回时回填给前端

### 7.3 它负责把结果回写到 JS Promise

代码位置：[MainActivity.kt:L158-L164](../src-tauri/gen/android/app/src/main/java/com/mde/app/MainActivity.kt#L158-L164)
```kt
private fun resolveCallback(callbackId: Long, value: String?) {
    if (callbackId <= 0L) return
    val wv = webViewRef ?: return
    val payload = if (value == null) "null" else "\"${escapeForJs(value)}\""
    val js = "if(window.__androidSafResolve){window.__androidSafResolve($callbackId, $payload);}"
    wv.post { wv.evaluateJavascript(js, null) }
}
```

也就是说，Android 原生并不是直接返回给 React，而是通过 WebView 注入的 JS 回调桥返回。

## 8. `SafBridge` 负责真正的 Android 文件操作

`MainActivity` 负责桥接和选择器，真正的 SAF 文件操作实现则在 [../src-tauri/gen/android/app/src/main/java/com/mde/app/SafBridge.kt](../src-tauri/gen/android/app/src/main/java/com/mde/app/SafBridge.kt#L18-L29)。

项目把它拆开，其实很合理：

- `MainActivity`
  - 管 Activity 生命周期、WebView 注入、picker 回调
- `SafBridge`
  - 管文件系统能力本身

这就是 Android 侧的职责分层。

## 10. 业务层是怎么真正“吃掉平台差异”的

真正把平台差异屏蔽到桥接层之上的，是 [../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L153-L175)。

它让业务动作依然保持统一语义，例如“打开文件”“保存文件”“打开目录”“资源管理器新建文件”。

比如打开目录时：

- 桌面端走 Tauri dialog
- Android 走 SAF picker
- 上层得到的仍然只是一个“目录句柄”

这种设计让 UI 组件不需要知道：

- 现在是桌面还是 Android
- 当前拿到的是磁盘路径还是 `content://`
- 对应读写到底落到 Rust 还是 Kotlin

因此真正的跨平台实现不是“UI 组件自己跨平台”，而是“业务层站在统一语义上工作，桥接层负责平台差异”。

## 11. 除了文件访问，项目还做了哪些跨平台适配

跨平台不是只有文件读写。

### 11.1 软键盘与安全区适配

[../src/hooks/useViewportInsets.js](../src/hooks/useViewportInsets.js#L18-L40) 负责把输入法遮挡和可视区域变化同步到 CSS 变量：

代码位置：[useViewportInsets.js:L30-L39](../src/hooks/useViewportInsets.js#L30-L39)
```js
const layoutHeight = window.innerHeight;
const visibleHeight = vv.height + vv.offsetTop;
const inset = Math.max(0, layoutHeight - visibleHeight);

document.documentElement.style.setProperty(KB_VAR, `${inset}px`);
if (inset > KB_OPEN_THRESHOLD) {
  document.documentElement.classList.add(KB_OPEN_CLASS);
} else {
  document.documentElement.classList.remove(KB_OPEN_CLASS);
}
```

这里的设计很关键：
它没有把软键盘高度塞进 React state，而是直接写到根元素 CSS 变量，避免高频视口变化导致整棵 React 树重渲染。

### 11.2 主题切换动画能力按平台降级

主题切换时，项目优先使用浏览器支持的 View Transition API，不支持时再回退。
相关逻辑在 [../src/components/layout/sidebar/Sidebar.jsx](../src/components/layout/sidebar/Sidebar.jsx#L400-L415)：

代码位置：[Sidebar.jsx:L400-L407](../src/components/layout/sidebar/Sidebar.jsx#L400-L407)
```js
function handleThemeSwitch() {
  const element = themeButtonRef.current;
  if (!element) { toggleTheme(); return; }
  if (!document.startViewTransition) { toggleTheme(); return; }
  const transition = document.startViewTransition(() => { toggleTheme(); });
```

这也是一种典型的跨平台策略：

- 优先用更高级能力
- 没有就优雅退化

Monaco 不是普通组件，它会在模块加载阶段缓存一部分界面文案。
所以项目专门写了 [../src/utils/monacoLocaleBoot.js](../src/utils/monacoLocaleBoot.js#L14-L16)，要求在真正导入 Monaco 之前先完成语言初始化：

## 13. 结论

当前项目的跨平台实现，本质上不是“到处判断平台”，而是分成了 3 个层次：

- 前端业务层
  - 只表达“我要打开、保存、搜索、列目录、显示到资源管理器”这些统一语义
- JS bridge 层
  - 用 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L1) 和 [../src/utils/androidSaf.js](../src/utils/androidSaf.js#L1) 把平台差异隐藏起来
- 原生执行层
  - 桌面端走 Rust command
  - Android 走 `MainActivity + SafBridge + SAF`

所以这个项目真正厉害的地方，不是某个平台的实现细节，而是它把“平台差异”压到了足够低的层级，让上面的文件管理、编辑器、设置等模块都能继续用统一语义工作。
