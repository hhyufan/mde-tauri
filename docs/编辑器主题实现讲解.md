# 编辑器主题实现讲解

## 1. 这篇文档讲什么

这篇文档虽然文件名叫“编辑器主题实现讲解”，但正文实际讲的是当前项目里编辑器主体实现，重点解释“Markdown 渲染阶段”以及参与编辑过程的前端工具链。  
范围按你的要求分成几条主线：

- Milkdown 所见即所得渲染
- Monaco + MarkdownPreview 的源码编辑 / 预览模式
- 所有内容主体相关呈现
- toolbar 工具栏
- 参与编辑过程的前端工具

这篇文档不从用户功能列表讲，而是从“正文如何在前端被组织、编辑、渲染、预览和增强”这条链路讲。

如果只先记一句话，可以把当前编辑器实现理解成：

`EditorContent 总调度 -> Monaco / Milkdown / MarkdownPreview 三条渲染路径 -> editorBuffer 作为正文总线 -> toolbar 与前端编辑增强工具接入`

---

## 2. 整体设计：项目不是一个编辑器包打天下

当前项目的编辑器区并不是只依赖一个组件，而是把 3 条不同职责的渲染路径组合起来：

- [../src/components/editor/MonacoEditor.jsx](../src/components/editor/MonacoEditor.jsx#L40-L475)
  - 负责源码编辑
- [../src/components/editor/MilkdownMarkdownEditor.jsx](../src/components/editor/MilkdownMarkdownEditor.jsx#L1826-L1865)
  - 负责所见即所得编辑
- [../src/components/editor/MarkdownPreview.jsx](../src/components/editor/MarkdownPreview.jsx#L265-L567)
  - 负责只读预览

真正决定“当前该渲染哪一种”的，是 [../src/components/layout/content/EditorContent.jsx](../src/components/layout/content/EditorContent.jsx#L182-L516)。

它先判断当前标签、视图模式和文件类型：

代码位置：[EditorContent.jsx:L184-L201](../src/components/layout/content/EditorContent.jsx#L184-L201)
```js

const activeTabId = useEditorStore((s) => s.activeTabId);

const activeTabMeta = useMemo(
  () => tabs.find((item) => item.id === activeTabId) || null,

const isMarkdown = /\.(md|markdown|mdx)$/i.test(activeTabMeta?.name || '');
```

然后在渲染阶段做分流：


这段代码已经把项目的编辑器主体结构讲清楚了：

- `edit`
  - 用 Monaco 纯源码编辑
- `preview`
  - 对 Markdown 文件走 Milkdown 所见即所得
- `split`
  - 左侧 Monaco 编辑，右侧 MarkdownPreview 预览
- 非 Markdown 文件
  - 即使切到 preview 或 split，也会回退成 Monaco

所以这个项目的编辑器体系，本质上是“调度层 + 多渲染器”的组合设计。

## 3. 正文真值并不直接放在 React 里，而是走 `editorBuffer`

这是整个编辑器实现最关键的设计点之一。  
如果你只看表面，会以为正文内容直接存在 React state 或 Zustand 里；但实际上，项目专门把高频正文输入从 React 状态树里抽离出来，放到了 [../src/utils/editorBuffer.js](../src/utils/editorBuffer.js#L1-L99)。

文件开头就直接说明了设计意图：

代码位置：[editorBuffer.js:L1-L11](../src/utils/editorBuffer.js#L1-L11)
```js
/**
 * 编辑器的内存缓冲区。
 *
 * 它刻意放在 React 状态之外，这样用户在 Monaco 中输入时就不会触发
 * Zustand 更新，也不会连带让编辑区、预览区、大纲、状态栏、标签栏等一起重渲染。
 */
```

它的核心实现非常简单：

代码位置：[editorBuffer.js:L13-L14](../src/utils/editorBuffer.js#L13-L14) + [editorBuffer.js:L49-L56](../src/utils/editorBuffer.js#L49-L56) + [editorBuffer.js:L61-L63](../src/utils/editorBuffer.js#L61-L63) + [editorBuffer.js:L96-L98](../src/utils/editorBuffer.js#L96-L98)（组合摘录）
```js
const buffers = new Map();
const listeners = new Set();

export function setBuffer(tabId, content) {
  if (!tabId) return;
  const cur = buffers.get(tabId);
  if (cur === content) return;
  buffers.set(tabId, content);
  pendingIds.add(tabId);
  schedule();
}

export function getBuffer(tabId, fallback = '') {
  return buffers.has(tabId) ? buffers.get(tabId) : fallback;
}

export function subscribe(fn) {
  listeners.add(fn);
  return () => listeners.delete(fn);
}
```

这说明项目把正文处理拆成了两层：

- 编辑器自己维护当前高频内容
- 其他消费者通过订阅缓冲区，按自己的节奏获取正文

对应的 React 订阅入口是 [../src/hooks/useEditorBufferContent.js](../src/hooks/useEditorBufferContent.js#L28-L57)：

代码位置：[useEditorBufferContent.js:L28-L54](../src/hooks/useEditorBufferContent.js#L28-L54)
```js
export function useEditorBufferContent(tabId, fallback = '', delay = 220) {
  const [content, setContent] = useState(() => getBuffer(tabId, fallback));

  useEffect(() => {
    let timer = null;
    const unsub = subscribe((ids) => {
      if (!ids.has(tabIdRef.current)) return;
      if (timer) clearTimeout(timer);
      const next = getBuffer(tabIdRef.current, fallbackRef.current);
      const actualDelay = pickDelay(delay, next?.length || 0);
      timer = setTimeout(() => {
        setContent(getBuffer(tabIdRef.current, fallbackRef.current));
      }, actualDelay);
    });
    return () => {
      unsub();
      if (timer) clearTimeout(timer);
    };
  }, [delay]);
```

这里还做了一个很重要的优化：  
文档越长，更新延迟越大，避免长文档下预览、大纲等重型消费者抢占输入线程。

所以整个编辑器主体不是“谁都盯着正文实时渲染”，而是：

- Monaco / Milkdown 负责写 buffer
- Preview / 大纲 / 其他消费者负责节流读取 buffer

## 4. Monaco 路径：源码编辑模式是怎么工作的

[../src/components/editor/MonacoEditor.jsx](../src/components/editor/MonacoEditor.jsx#L40-L475) 负责源码编辑。  
它不是简单包一层 Monaco，而是把主题、语言、输入节流、自动保存、移动端增强都包进来了。

创建实例时，项目把编辑器的主要表现配置一次性注入：

这里已经能看出几个设计取向：

- 关闭原生 `contextmenu`，说明项目要接管右键菜单
- 字体、行高、换行、行号等配置直接与项目设置联动

### 4.2 Monaco 输入后如何进入正文总线

Monaco 输入不是每打一个字就把整篇内容写进 Zustand，而是先写 `editorBuffer`，再分发其他副作用：

代码位置：[MonacoEditor.jsx:L222-L233](../src/components/editor/MonacoEditor.jsx#L222-L233)
```js
editor.onDidChangeModelContent(() => {
  if (suppressModelChangeRef.current) return;
  const tabId = currentTabIdRef.current;
  if (!tabId) return;
  const value = editor.getValue();
  setBuffer(tabId, value);
  scheduleDirtyMark();
  scheduleCharCount();
  scheduleAutoSave();
});
```

副作用本身都做了节流：

代码位置：[MonacoEditor.jsx:L186-L194](../src/components/editor/MonacoEditor.jsx#L186-L194) + [MonacoEditor.jsx:L213-L220](../src/components/editor/MonacoEditor.jsx#L213-L220)（组合摘录）
```js
const scheduleDirtyMark = () => {
  if (dirtyMarkScheduled) return;
  dirtyMarkScheduled = true;
  setTimeout(() => {
    dirtyMarkScheduled = false;
    const tabId = currentTabIdRef.current;
    if (tabId) markTabDirty(tabId, true);
  }, 200);
};

const scheduleAutoSave = () => {
  if (autoSaveScheduled) return;
  autoSaveScheduled = true;
  setTimeout(() => {
    autoSaveScheduled = false;
    onAutoSaveRef.current?.();
  }, 300);
};
```

这说明 Monaco 路径的真实处理顺序是：

1. 编辑器输入
2. 立即写 `editorBuffer`
3. 延迟标记 dirty
4. 延迟更新字数统计
5. 延迟触发自动保存

这就是为什么输入体验可以保持顺滑，而预览、大纲、状态栏又能同步更新。

### 4.3 Monaco 如何接 toolbar

Monaco 暴露了一个 imperative API 给上层使用：

代码位置：[MonacoEditor.jsx:L96-L131](../src/components/editor/MonacoEditor.jsx#L96-L131)
```js
useImperativeHandle(ref, () => ({
  getEditor: () => editorRef.current,
  getCurrentValue: () => editorRef.current?.getValue() ?? '',
  insertText(text) {
    const selection = editor.getSelection();
    editor.executeEdits('toolbar', [{
      range: selection,
      text,
      forceMoveMarkers: true,
    }]);
    editor.focus();
  },
  wrapSelection(before, after) {
    const selection = editor.getSelection();
    const selectedText = editor.getModel().getValueInRange(selection);
    editor.executeEdits('toolbar', [{
      range: selection,
      text: before + selectedText + after,
      forceMoveMarkers: true,
    }]);
    editor.focus();
  },
}), []);
```

因此 toolbar 对 Monaco 而言，本质上是“调用编辑器 ref 暴露的插入/包裹动作”。

## 5. Milkdown 路径：所见即所得渲染是怎么工作的

[../src/components/editor/MilkdownMarkdownEditor.jsx](../src/components/editor/MilkdownMarkdownEditor.jsx#L1826-L1865) 对应所见即所得模式。  
它不是一个单纯的 Markdown viewer，而是真正可编辑的 WYSIWYG 编辑器。

### 5.1 Milkdown 初始化阶段

编辑器创建时会挂载一整组插件：

### 5.2 Milkdown 编辑后如何回写 Markdown

Milkdown 虽然是所见即所得，但整个系统的共享正文仍然是 Markdown 文本。  
所以它最关键的一步就是：编辑完结构化文档后，把最新 Markdown 回写到正文总线。

核心逻辑就在 `markdownUpdated` 里：

代码位置：[MilkdownMarkdownEditor.jsx:L1437-L1442](../src/components/editor/MilkdownMarkdownEditor.jsx#L1437-L1442)
```js
ctx.get(listenerCtx)
  .markdownUpdated((_, markdown, prevMarkdown) => {
    if (markdown === prevMarkdown) return;
    scheduleEditorSideEffects(markdown);
    setRenderTick((value) => value + 1);
  });
```

也就是说，Milkdown 不是系统的“第二套文档真值”，它只是另一种编辑 UI。  
项目真正统一交换的仍然是 Markdown。

### 5.3 Milkdown 如何接 toolbar

Milkdown 也暴露了统一动作接口：

代码位置：[MilkdownMarkdownEditor.jsx:L1456-L1532](../src/components/editor/MilkdownMarkdownEditor.jsx#L1456-L1532)
```js
useImperativeHandle(ref, () => ({
  handleToolbarAction(action) {
    let handled = false;
    editorRef.current?.action((ctx) => {
      const commands = ctx.get(commandsCtx);
      const view = ctx.get(editorViewCtx);
      const run = (command, payload) => {
        handled = Boolean(commands.call(command.key, payload));
        view.focus();
      };

      switch (action?.command) {
        case 'bold':
          run(toggleStrongCommand);
          break;
        case 'italic':
          handled = toggleMark(emphasisSchema.type(ctx), { marker: '*' })(view.state, view.dispatch.bind(view));
          view.focus();
          break;
        case 'table':
          run(insertTableCommand, { row: 3, col: 3 });
          break;
        case 'image':
          run(insertImageCommand, { src: 'url', alt: 'alt', title: '' });
          break;
      }
    });
    return handled;
  },
  insertText(text) {
    editorRef.current?.action(insert(text, true));
  },
  wrapSelection(before, after) {
    editorRef.current?.action((ctx) => {
      const view = ctx.get(editorViewCtx);
      const { from, to, empty } = view.state.selection;
      if (empty) {
        insert(`${before}${after}`, true)(ctx);
        return;
      }
      const selected = getMarkdown({ from, to })(ctx);
      replaceRange(`${before}${selected}${after}`, { from, to })(ctx);
    });
  },
}), [content]);
```

这说明项目在 Monaco 和 Milkdown 上做了一个很重要的统一抽象：  
**都向上暴露 `handleToolbarAction / insertText / wrapSelection` 这组协议。**

这样上层 `EditorContent` 就不需要知道“当前是 Monaco 还是 Milkdown”，只需要发 toolbar 动作。

## 6. MarkdownPreview 路径：源码预览模式是怎么渲染的

分屏模式下，左侧是 Monaco，右侧是 [../src/components/editor/MarkdownPreview.jsx](../src/components/editor/MarkdownPreview.jsx#L265-L567)。  
这个组件负责只读渲染。

### 6.1 预览的内容来源

预览不是直接读 tab 的静态内容，而是订阅 `editorBuffer`：

### 6.4 代码块、Mermaid、图片这些主体内容如何呈现

#### 6.4.1 代码块与 Mermaid

代码节点会先分流：

代码位置：[MarkdownPreview.jsx:L159-L174](../src/components/editor/MarkdownPreview.jsx#L159-L174)
```js
function CodeBlock({ children, className, ...props }) {
  const match = /language-(\w+)/.exec(className || '');
  const lang = match ? match[1] : '';
  const code = String(children).replace(/\n$/, '');

  if (lang === 'mermaid') {
    const isDark = document.documentElement.dataset.theme === 'dark';
    return <MermaidRenderer code={code} isDark={isDark} />;
  }

  if (!match) {
    return <code className={className} {...props}>{children}</code>;
  }

  return <code className={`language-${lang}`} {...props}>{children}</code>;
}
```

这意味着：

- 普通行内代码
  - 直接渲染 `<code>`
- 普通代码块
  - 挂语言类名，后续交给 Prism 高亮
- `language-mermaid`
  - 走专门的 [../src/components/editor/MermaidRenderer.jsx](../src/components/editor/MermaidRenderer.jsx#L23-L76)

#### 6.4.2 代码高亮不是同步整页跑，而是空闲分批处理

预览层没有在 React render 里直接高亮，而是把 Prism 放到浏览器空闲时间执行：

代码位置：[MarkdownPreview.jsx:L370-L414](../src/components/editor/MarkdownPreview.jsx#L370-L414)
```js
const scheduleHighlight = useCallback(() => {
  const pending = [];
  container.querySelectorAll('pre > code[class*="language-"]').forEach((codeEl) => {
    if (codeEl.querySelector(':scope > .token')) return;
    if (codeEl.classList.contains('language-mermaid')) return;
    pending.push(codeEl);
  });

  let i = 0;
  const work = (deadline) => {
    while (i < pending.length && hasTime()) {
      const el = pending[i++];
      try {
        Prism.highlightElement(el);
      } catch (_) {}
    }

    if (i < pending.length) {
      idleJobRef.current = requestIdle(work, { timeout: 500 });
    } else {
      addLangTags(container);
    }
  };
}, [addLangTags, cancelHighlightJob]);
```

这说明预览层同样在避免“长文档一更新就整页重活同步执行”。

#### 6.4.3 图片渲染

图片节点也不是直接把 Markdown 里的 `src` 原样塞给 `<img>`，而是走路径解析和对象 URL 生命周期管理：

代码位置：[MarkdownPreview.jsx:L200-L224](../src/components/editor/MarkdownPreview.jsx#L200-L224)
```js
function MarkdownImage({ src, alt, documentPath, ...props }) {
  const [resolvedSrc, setResolvedSrc] = useState(src);

  useEffect(() => {
    let objectUrl = null;
    let cancelled = false;

    setResolvedSrc(src);
    loadMarkdownImageSrc(src, documentPath).then((result) => {
      if (cancelled) {
        if (result.objectUrl) URL.revokeObjectURL(result.objectUrl);
        return;
      }
      objectUrl = result.objectUrl;
      setResolvedSrc(result.src || src);
    });

    return () => {
      cancelled = true;
      if (objectUrl) URL.revokeObjectURL(objectUrl);
    };
  }, [documentPath, src]);

  return <img src={resolvedSrc} alt={alt || ''} loading="lazy" {...props} />;
}
```

### 8.3 `EditorContent` 如何把 toolbar 动作交给当前编辑器

真正的分发点在 [../src/components/layout/content/EditorContent.jsx](../src/components/layout/content/EditorContent.jsx#L212-L225)：

代码位置：[EditorContent.jsx:L212-L225](../src/components/layout/content/EditorContent.jsx#L212-L225)
```js
const handleToolbarInsert = useCallback((action) => {
  const editor = monacoRef.current;
  if (!editor) return;

  if (typeof editor.handleToolbarAction === 'function' && editor.handleToolbarAction(action)) {
    return;
  }

  if (action.type === 'insert') {
    editor.insertText(action.text);
  } else if (action.type === 'wrap') {
    editor.wrapSelection(action.before, action.after);
  }
}, []);
```

这就是整个工具栏统一协议的落点：

- 当前编辑器若能理解“高级命令”
  - 优先走 `handleToolbarAction`
- 否则
  - 回退到通用 `insertText / wrapSelection`

这套机制让 Monaco 和 Milkdown 可以共用同一套 toolbar。

### 9.1 Monaco 自定义右键菜单

```js
  browserEvent?.stopPropagation?.();

  const selection = editor.getSelection();
```

代码位置：[MobileSelectionBar.jsx:L242-L254](../src/components/editor/MobileSelectionBar.jsx#L242-L254)
```js
const handleCopy = async (e) => {
  e?.preventDefault?.();
  e?.stopPropagation?.();
  const text = getSelectionText();
  if (!text) return;
```

## 10. 自动保存是如何接进三条编辑链路的

编辑器主体里还有一个必须讲清楚的点，就是自动保存。  
因为它连接了“正文输入”和“文件管理落盘”。

在 [../src/components/layout/content/EditorContent.jsx](../src/components/layout/content/EditorContent.jsx#L458-L475) 中，Monaco 和 Milkdown 都收到了同一个 `onAutoSave`：

代码位置：[EditorContent.jsx:L459-L464](../src/components/layout/content/EditorContent.jsx#L459-L464) + [EditorContent.jsx:L468-L473](../src/components/layout/content/EditorContent.jsx#L468-L473)（组合摘录）

- 编辑器负责判断“现在适合触发自动保存”
- 文件管理层负责决定“怎么保存”

这也体现出编辑器主体和文件管理之间的解耦。

## 11. 为什么这个项目要同时保留 Milkdown 和 Monaco

如果你把整个实现串起来，会发现这不是重复造轮子，而是职责分工。

### 11.1 Monaco 适合源码真编辑

它的优点是：

- 纯文本编辑稳定
- 可控性高
- 非 Markdown 文件也能统一承载
- 搜索、选区、输入、快捷操作成熟

### 11.2 Milkdown 适合所见即所得

它的优点是：

- 对结构化块操作友好
- 表格、任务列表、引用块等更接近“文档编辑器”体验
- 工具栏动作可以直接变成文档命令

### 11.3 MarkdownPreview 适合只读高质量预览

它的优点是：

- 不承载编辑行为，职责单纯
- 可以放心做重型渲染增强
- 更适合高亮、Mermaid、脚注、表格、图片、链接、锚点等只读呈现逻辑

所以三者并不是互相替代，而是：

- Monaco
  - 负责源码编辑
- Milkdown
  - 负责所见即所得编辑
- MarkdownPreview
  - 负责预览呈现

## 13. 结论

当前项目的编辑器主体实现，不是单个 Markdown 编辑器组件，而是一套围绕正文总线构建的多路径渲染系统：

- `EditorContent` 负责总调度，决定当前走 Monaco、Milkdown 还是 MarkdownPreview
- `editorBuffer` 负责承接高频正文输入，避免 React 和 Zustand 被每次击键穿透
- `MonacoEditor` 负责源码编辑路径
- `MilkdownMarkdownEditor` 负责所见即所得路径
- `MarkdownPreview` 负责只读预览路径
- `FloatingToolbar` 负责统一的 Markdown 操作入口
- `MobileSelectionBar`、大纲跳转、代码块增强等共同构成编辑辅助工具链

如果从“渲染阶段”这个角度总结，它的核心思路就是：

- 把正文输入和 React 渲染解耦
- 把编辑和预览分成不同渲染器
- 用统一动作协议把工具栏接入不同编辑器
- 把代码高亮、Mermaid、数学公式、图片、脚注等重型内容呈现下沉到专门层处理

所以这套实现真正厉害的地方，不是某个单独组件，而是它把源码编辑、所见即所得、只读预览和编辑辅助工具组合成了一条完整的 Markdown 编辑链路。
