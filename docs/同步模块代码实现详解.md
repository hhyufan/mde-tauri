这篇文档不是只解释“同步模块做了什么”，而是帮助你顺着当前项目代码，真正吃透“同步这一块的服务是怎么实现的”。  
阅读目标是让你能够回答下面这些问题：

- 同步能力的总入口在哪里

- 为什么前端要维护 `fileId`、队列、冲突列表、cursor
- 前端请求是怎么进入后端 `/sync/*` 接口的
- 后端如何基于 `baseRev`、`mutationId` 做并发控制和幂等
- 一次完整的“保存 -> 入队 -> 推送 -> 拉取 -> 冲突处理”到底是怎么跑通的

如果你要彻底吃透这一块，建议先读这篇文档，再按文中的阅读顺序去点代码。

## 2. 先建立整体视角

同步模块不是单个文件，而是一整套跨前端和后端的协作系统。  
在当前项目里，它可以粗分为五层：

1. 调用入口层
   - 例如应用启动、保存文件、打开云文档、添加书签、删除文档、同步配置
2. 前端同步服务层
   - 主要是 [../src/services/syncEngine.js](../src/services/syncEngine.js#L189-L1178)
3. 前端状态存储层
   - 主要是 [../src/store/useSyncStore.js](../src/store/useSyncStore.js#L71-L390)
   - 辅助还有 [../src/store/useFileIdStore.js](../src/store/useFileIdStore.js#L34-L149) 与 [../src/store/useExternalDocsStore.js](../src/store/useExternalDocsStore.js#L36-L118)
4. 后端接口与业务层
   - [../mde-server/src/sync/sync.controller.ts](../mde-server/src/sync/sync.controller.ts#L28-L67)
   - [../mde-server/src/sync/sync.service.ts](../mde-server/src/sync/sync.service.ts#L171-L429)
5. 后端数据模型层
   - [../mde-server/src/schemas/sync-document.schema.ts](../mde-server/src/schemas/sync-document.schema.ts#L13-L92)
   - [../mde-server/src/schemas/sync-config.schema.ts](../mde-server/src/schemas/sync-config.schema.ts#L5-L68)

可以把它理解成下面这条链路：

`UI/调用方 -> syncEngine -> store -> apiClient -> SyncController -> SyncService -> Mongo schema`

这也是后面讲解的主线。

## 3. 前端同步总入口：`syncEngine`

如果只能先看一个文件，那就先看 [../src/services/syncEngine.js](../src/services/syncEngine.js#L189-L222)。  
它是整个同步模块的总调度中心，几乎所有同步能力都会收束到这里。

从类定义开始就能看出它的定位：

代码位置：[syncEngine.js:L189-L198](../src/services/syncEngine.js#L189-L198)
```js
class SyncEngine {
  constructor() {
    this.status = 'idle';
    this.listeners = new Set();
    this.retryTimer = null;
    this.syncing = false;
    this.configSyncTimer = null;
    this.suppressConfigAutoSync = false;
    this.setupConfigSubscriptions();
  }
}
```

这里有几个非常关键的状态：

- `status`
  - 表示当前同步状态，例如 `idle`、`syncing`、`synced`、`conflict`
- `retryTimer`
  - 控制失败后的延迟重试
- `syncing`
  - 避免并发执行多次完整同步
- `configSyncTimer`
  - 控制配置同步节流
- `setupConfigSubscriptions()`
  - 说明它不仅处理文档同步，也处理配置镜像同步

也就是说，`syncEngine` 不是某一个 API 的包装器，而是“同步总线”。

## 4. 同步模块到底管理了哪些对象

要读懂同步实现，先不要急着看网络请求，而要先搞清楚它到底在同步什么。

### 4.1 本地真实文件

本地真实文件是指当前设备上的真实路径文件，例如某个 `D:\notes\a.md`。  
这类文件会通过 [../src/store/useFileIdStore.js](../src/store/useFileIdStore.js#L40-L67) 绑定一个稳定的 `fileId`，从而在云端拥有统一身份。

在 [../src/services/syncEngine.js](../src/services/syncEngine.js#L347-L372) 里，这个过程由 `registerLocalDocument()` 负责：

代码位置：[syncEngine.js:L347-L372](../src/services/syncEngine.js#L347-L372)
```js
registerLocalDocument(filePath, meta = {}) {
  if (!filePath || isCloudPath(filePath)) return null;
  const fileIdStore = useFileIdStore.getState();
  const existingFileId = fileIdStore.idOf(filePath);
  const isBookmarked = useFileStore.getState().isBookmarked(filePath);

  let fileId = meta.fileId || existingFileId;
  if (!fileId && isBookmarked) {
    fileId = fileIdStore.getOrCreate(filePath);
  }

  if (!fileId) return null;

  fileIdStore.bind(filePath, fileId);
  useSyncStore.getState().bindLocalPath(fileId, filePath, {
    name: meta.name || basename(filePath),
    ext: meta.ext || extOf(meta.name || filePath),
    encoding: meta.encoding || 'UTF-8',
    lineEnding: meta.lineEnding || 'LF',
    source: meta.source || 'local',
    status: meta.status || 'idle',
    deleted: false,
  });
  return fileId;
}
```

这段代码说明：

- 本地文件不是天然进入云同步
- 必须先获得 `fileId`
- 然后在 `useSyncStore` 中建立文档镜像

项目设计上把“本地路径”和“云端身份”刻意分离了。

### 4.2 external 云文档

external 文档指的是“云端有，但当前设备上还没有真实落盘路径”的文档。  
它们不直接写入本地文件，而是暂存在 [../src/store/useExternalDocsStore.js](../src/store/useExternalDocsStore.js#L42-L61)。

在 [../src/services/syncEngine.js](../src/services/syncEngine.js#L773-L819) 中，无法写回本地路径时会退回 external 模式：

代码位置：[syncEngine.js:L802-L811](../src/services/syncEngine.js#L802-L811)
```js
useExternalDocsStore.getState().put(doc.fileId, {
  name: doc.fileName || basename(doc.originalPath || '') || doc.fileId,
  ext: extOf(doc.fileName || doc.originalPath || ''),
  encoding: doc.encoding || 'UTF-8',
  lineEnding: doc.lineEnding || 'LF',
  originalPath: doc.originalPath || '',
  content: decoded,
  checksum: doc.contentHash || doc.checksum || '',
  rev: doc.rev || 0,
});
```

这意味着同步系统并不要求“每份云文档在每台设备上都有真实文件路径”。  
这也是它能支持 `cloud://<fileId>` 逻辑路径的重要基础。

### 4.3 配置镜像

同步模块不仅同步文档，也同步编辑器配置。  
这部分逻辑也在 [../src/services/syncEngine.js](../src/services/syncEngine.js#L206-L234) 中，由 `setupConfigSubscriptions()` 和 `syncConfig()` 驱动。

例如：

代码位置：[syncEngine.js:L206-L222](../src/services/syncEngine.js#L206-L222)
```js
setupConfigSubscriptions() {
  useConfigStore.subscribe((state, prev) => {
    if ((state.configUpdatedAt || 0) !== (prev.configUpdatedAt || 0)) {
      this.scheduleConfigSync();
    }
  });
  useThemeStore.subscribe((state, prev) => {
    if ((state.themeUpdatedAt || 0) !== (prev.themeUpdatedAt || 0)) {
      this.scheduleConfigSync();
    }
  });
  useEditorStore.subscribe((state, prev) => {
    if ((state.uiStateUpdatedAt || 0) !== (prev.uiStateUpdatedAt || 0)) {
      this.scheduleConfigSync();
    }
  });
}
```

这说明配置同步并不是单独的服务，而是被纳入同一个同步引擎统一调度。

## 5. 前端状态中心：`useSyncStore`

如果 `syncEngine` 是“大脑”，那 [../src/store/useSyncStore.js](../src/store/useSyncStore.js#L71-L390) 就是“工作记忆”。  
它保存了同步过程中最核心的状态：

- `docs`
- `queue`
- `conflicts`
- `cursor`
- `lastSyncError`

初始化定义在这里：

代码位置：[useSyncStore.js:L71-L82](../src/store/useSyncStore.js#L71-L82)
```js
(set, get) => ({
  protocolVersion: SYNC_PROTOCOL_VERSION,
  localResetDone: false,
  localResetDoneScopes: {},
  docs: {},
  queue: [],
  conflicts: [],
  cursor: '',
  cursors: {},
  lastSyncError: null,
```

### 5.1 `docs`：本地同步镜像

`docs` 保存的是“前端对每个云文档当前状态的理解”，不是完整正文数据库。  
这个镜像里最关键的字段有：

- `fileId`
- `rev`
- `lastKnownServerRev`
- `localPath`
- `status`
- `deleted`

这些字段由 `upsertDoc()` 统一维护：

代码位置：[useSyncStore.js:L152-L173](../src/store/useSyncStore.js#L152-L173)
```js
upsertDoc: (fileId, patch) => {
  if (!fileId) return;
  const key = scopedDocKey(fileId);
  set((state) => ({
    docs: {
      ...state.docs,
      [key]: {
        ...(state.docs[key] || {
          fileId,
          rev: 0,
          lastKnownServerRev: 0,
          deleted: false,
          status: 'idle',
          ownerUserId: getCurrentUserScopeId(),
        }),
        ...patch,
        fileId,
        ownerUserId: getCurrentUserScopeId(),
      },
    },
  }));
}
```

你可以把它理解成“前端自己的同步索引”。

### 5.2 `queue`：待同步 mutation 队列

所有待推送操作都不会立刻散落到多个地方，而是先进入统一队列。  
入队逻辑在 `enqueueMutation()` 中：

代码位置：[useSyncStore.js:L244-L283](../src/store/useSyncStore.js#L244-L283)
```js
enqueueMutation: ({ fileId, type, payload, baseRev = 0, mutationId, dedupeKey }) => {
  if (!fileId || !type) return null;
  const id = mutationId || newMutationId();
  set((state) => {
    const ownerUserId = getCurrentUserScopeId();
    let nextQueue = [...state.queue];

    if (type === 'delete') {
      nextQueue = nextQueue.filter(
        (item) => !(isOwnedByUser(item?.ownerUserId, ownerUserId) && item.fileId === fileId)
      );
    } else if (dedupeKey) {
      nextQueue = nextQueue.filter(
        (item) => !(
          isOwnedByUser(item?.ownerUserId, ownerUserId)
          && item.fileId === fileId
          && item.dedupeKey === dedupeKey
        ),
      );
    }

    nextQueue.push({
      id,
      mutationId: id,
      fileId,
      type,
      payload,
      baseRev,
      retryCount: 0,
      nextRetryAt: Date.now(),
      status: 'pending',
      dedupeKey: dedupeKey || type,
      ownerUserId,
    });
    return { queue: nextQueue };
  });
  return id;
}
```

这里一口气体现出三套关键机制：

- `baseRev`
  - 本次变更基于哪个服务端版本产生
- `mutationId`
  - 本次操作的全局唯一 ID，用于幂等
- `dedupeKey`
  - 同一文件的同类快照型操作只保留最新一条

如果你读懂这个函数，整个同步队列模型就懂了一半。

### 5.3 `conflicts`：冲突列表

当前项目不是在冲突时立刻合并，而是把冲突对象加入 `conflicts` 列表，交给 UI 决策。

代码如下：

代码位置：[useSyncStore.js:L348-L358](../src/store/useSyncStore.js#L348-L358)
```js
addConflict: (conflict) => {
  if (!conflict?.fileId) return;
  const ownerUserId = getCurrentUserScopeId();
  set((state) => {
    const existing = state.conflicts.filter((item) => !(
      item.fileId === conflict.fileId
      && isOwnedByUser(item?.ownerUserId, ownerUserId)
    ));
    return { conflicts: [...existing, { ...conflict, ownerUserId }] };
  });
}
```

这说明冲突在前端被视为一种显式状态，而不是同步异常里的临时分支。

## 6. 文档是如何从“保存”进入同步系统的

真正理解同步，必须搞清楚“一个编辑过的文档是怎么进入同步系统的”。

### 6.1 本地文件保存后进入队列

本地文件保存后，`syncEngine` 会通过 `queueLocalUpsert()` 把最新内容送入同步队列。  
这段逻辑位于 [../src/services/syncEngine.js](../src/services/syncEngine.js#L377-L459)：

代码位置：[syncEngine.js:L377-L403](../src/services/syncEngine.js#L377-L403)
```js
async queueLocalUpsert(filePath, content, encoding = 'UTF-8', options = {}) {
  if (!filePath || isCloudPath(filePath)) {
    return { ok: false, reason: 'invalid-path' };
  }
  if (typeof content !== 'string') {
    return { ok: false, reason: 'missing-content' };
  }

  const existingFileId = useFileIdStore.getState().idOf(filePath);
  const isBookmarked = useFileStore.getState().isBookmarked(filePath);

  if (!existingFileId && !isBookmarked) {
    return { ok: true, skipped: 'not-bookmarked-and-not-cloud' };
  }

  const fileId = this.registerLocalDocument(filePath, {
    name: options.name || basename(filePath),
    encoding,
    lineEnding: options.lineEnding || 'LF',
    source: options.source || 'local',
  });
```

从这里能看出两个关键设计点：

1. 不是所有本地文件都会自动上云
   - 至少需要“已经绑定过 `fileId`”或“处于已收藏状态”
2. 入队前会先确保文档已经完成本地路径到 `fileId` 的注册

### 6.2 内容会先编码，再入队

继续往下读会发现，正文不是原样直接提交，而是先走 `encodeBody()`：

代码位置：[syncEngine.js:L412-L419](../src/services/syncEngine.js#L412-L419)
```js
const body = await encodeBody(content);
if (body.compressedBytes > MAX_REQUEST_BYTES) {
  return {
    ok: false,
    reason: 'too-large',
    size: body.size,
    compressed: body.compressedBytes,
  };
}
```

也就是说，前端会在入队前先完成：

- SHA-256 校验
- 小文件明文 / 大文件 gzip
- 请求体积限制控制

这能解释为什么同步引擎文件里同时有压缩、校验和编码逻辑。

### 6.3 队列里的 `baseRev` 是怎么来的

入队时，`baseRev` 不是随便写的，而是基于当前文档镜像里的版本信息：

代码位置：[syncEngine.js:L435-L453](../src/services/syncEngine.js#L435-L453)
```js
useSyncStore.getState().enqueueMutation({
  fileId,
  type: 'upsert',
  baseRev: doc?.lastKnownServerRev || doc?.rev || 0,
  dedupeKey: 'upsert',
  payload: {
    fileName: options.name || basename(filePath),
    originalPath: filePath,
    source: options.source || 'local',
    content: body.content,
    rawContent: content,
```

这说明前端的同步写入并不是“当前内容直接发上去”，而是“基于我已知的服务端版本发上去”。

## 7. 完整同步流程 `fullSync()` 是怎么编排的

真正的完整同步，不是只有“推送”或者“拉取”，而是一个有固定顺序的流程。  
这个顺序都在 [../src/services/syncEngine.js](../src/services/syncEngine.js#L1153-L1178) 的 `fullSync()` 里：

代码位置：[syncEngine.js:L1153-L1169](../src/services/syncEngine.js#L1153-L1169)
```js
async fullSync() {
  if (!useAuthStore.getState().isLoggedIn || !useConfigStore.getState().syncEnabled) {
    return;
  }
  if (this.syncing) return;
  this.syncing = true;
  this.setStatus('syncing');
  try {
    await this.ensureLocalReset();
    await this.ensureRemoteProtocol();
    await this.syncConfig();
    const queueOk = await this.processQueue();
    if (queueOk) {
      await this.pullRemoteChanges();
      await this.syncConfig();
    }
```

这个顺序非常重要：

1. `ensureLocalReset()`
   - 处理账号切换后的本地作用域重置
2. `ensureRemoteProtocol()`
   - 保证前后端同步协议版本兼容
3. `syncConfig()`
   - 先对齐配置镜像
4. `processQueue()`
   - 先把本地待推送 mutation 送出去
5. `pullRemoteChanges()`
   - 再拉取远端增量变更
6. 再次 `syncConfig()`
   - 避免旧快照回写配置

这套顺序体现了当前设计优先级：  
**优先保留本地最新编辑意图，再吸收远端变更。**

前端并不是直接用 `fetch` 到处调接口，而是统一通过 [../src/services/apiClient.js](../src/services/apiClient.js#L18-L110)。

代码位置：[apiClient.js:L18-L18](../src/services/apiClient.js#L18-L18) + [apiClient.js:L70-L72](../src/services/apiClient.js#L70-L72)（当前实现已调整）

```js
const apiClient = axios.create({

```

然后在请求拦截器里自动注入 JWT：

## 9. 后端 `SyncService` 是整个服务端同步核心

前端的 `syncEngine` 对应到后端，就是 [../mde-server/src/sync/sync.service.ts](../mde-server/src/sync/sync.service.ts#L171-L429)。  
如果你想真正吃透服务端同步，最应该精读的就是这个文件。

### 9.1 `pushFile()`：写入文件的核心

这是后端最关键的入口之一。  
它负责同时处理：

- 新建文档
- 更新文档
- 幂等重试
- 版本冲突
- 设备路径绑定

核心代码如下：

代码位置：[sync.service.ts:L222-L231](../mde-server/src/sync/sync.service.ts#L222-L231)
```ts
async pushFile(userId: string, fileId: string, doc: PushFileDto) {
  const userObjectId = this.userObjectId(userId);
  const safeDeviceId = this.sanitizeDeviceId(doc.deviceId);
  const existing = await this.docModel
    .findOne({ userId: userObjectId, fileId })
    .lean<any>();

  if (existing?.lastMutationId && existing.lastMutationId === doc.mutationId) {
    return this.serializeDoc(existing);
  }
```

第一步就不是写数据库，而是先处理幂等。

然后是版本判定：

代码位置：[sync.service.ts:L255-L271](../mde-server/src/sync/sync.service.ts#L255-L271)
```ts
if (!existing) {
  if ((doc.baseRev ?? 0) !== 0) {
    this.throwRevisionConflict(null);
  }
}

if ((doc.baseRev ?? 0) !== existing.rev) {
  this.throwRevisionConflict(existing);
}
```

成功更新时则用 `$inc: { rev: 1 }` 推进版本：

```ts
代码位置：[sync.service.ts:L274-L285](../mde-server/src/sync/sync.service.ts#L274-L285)
const updated = await this.docModel.findOneAndUpdate(
  {
    userId: userObjectId,
    fileId,
    rev: existing.rev,
  },
  {
    $set: payload,
    $inc: { rev: 1 },
  },
  { new: true },
).lean<any>();
```

这已经基本体现了后端同步写入的全部核心思想。

### 9.2 `getChanges()`：增量拉取入口

前端不是每次都把所有文档全量拉回，而是依赖 cursor 增量拉取。  
这部分由 `getChanges()` 负责：

代码位置：[sync.service.ts:L196-L204](../mde-server/src/sync/sync.service.ts#L196-L204)
```ts
async getChanges(userId: string, since?: string) {
  const sinceDate = since ? new Date(since) : null;
  const filter: Record<string, any> = {
    userId: this.userObjectId(userId),
    fileId: { $exists: true, $nin: [null, ''] },
  };
  if (sinceDate && !Number.isNaN(sinceDate.getTime())) {
    filter.updatedAt = { $gt: sinceDate };
  }
```

这里可以看出，增量同步依赖的是：

- `updatedAt`
- `cursor`
- 文档元数据清单

而不是基于操作日志逐条回放。

### 9.3 `deleteFile()`：删除不是物理删除，而是墓碑模型

删除逻辑也很值得注意。  
当前项目不是简单 `remove`，而是“墓碑删除”：

代码位置：[sync.service.ts:L395-L406](../mde-server/src/sync/sync.service.ts#L395-L406)
```ts
const updated = await this.docModel.findOneAndUpdate(
  { userId: userObjectId, fileId, rev: existing.rev },
  {
    $set: {
      deleted: true,
      deletedAt: new Date(),
      lastMutationId: dto.mutationId,
    },
    $inc: { rev: 1 },
  },
  { new: true },
).lean<any>();
```

这样做的意义是：

- 其他设备能够感知删除事件
- 删除本身也具有版本号
- 删除冲突可以被前端显式处理

这也是为什么前端在处理远端删除时仍然会先进入冲突判断。

### 9.4 `bindPath()`：设备路径映射

同步模块不是只同步正文，还同步“某个云文档在当前设备上对应哪个本地路径”。  
这部分在 `bindPath()` 中实现：

代码位置：[sync.service.ts:L363-L372](../mde-server/src/sync/sync.service.ts#L363-L372)
```ts
const updated = await this.docModel.findOneAndUpdate(
  { userId: userObjectId, fileId, deleted: false },
  {
    $set: {

```

也就是说，服务端文档模型里不仅有正文和版本，还有：

- `deviceBindings`
- `lastMutationId`
- `deleted/deletedAt`

同步文档本质上是一个“带设备关系的版本化文档对象”。

## 10. 冲突处理为什么是当前同步模块的核心难点

如果你想“彻底吃透”同步代码，那最该重点啃的不是普通推送，而是冲突处理。  
因为这一部分同时跨越了前端实时状态、队列、后端版本控制和用户决策。

### 10.1 前端如何判定要不要升级为冲突

判断逻辑在 [../src/services/syncEngine.js](../src/services/syncEngine.js#L639-L650) 的 `getRemoteConflictDecision()`：

代码位置：[syncEngine.js:L639-L650](../src/services/syncEngine.js#L639-L650)
```js
getRemoteConflictDecision(fileId, remoteContent, localPath = '', { force = false } = {}) {
  const localState = this.getOpenLocalState(fileId, localPath);
  if (force || useConfigStore.getState().autoSave) {
    return { shouldConflict: false, localState };
  }
  if (!localState.modified) {
    return { shouldConflict: false, localState };
  }
  return {
    shouldConflict: localState.content !== (remoteContent || ''),
    localState,
  };
}
```

这说明冲突判断不是只看版本号，而是同时看：

- 是否强制接受远端
- 是否开启自动保存
- 标签页是否仍有未保存修改
- 本地内容和远端内容是否真的不同

### 10.2 服务端返回 409 后，前端怎么做

对应逻辑在 `processMutation()` 的 `catch 409` 分支：

代码位置：[syncEngine.js:L953-L962](../src/services/syncEngine.js#L953-L962)
```js
if (err?.response?.status === 409) {
  const current = err.response?.data?.current;
  let remoteDoc = current;
  if (current?.fileId && !current.content && !current.deleted) {
    const { data } = await apiClient.get(`/sync/file/${encodeURIComponent(current.fileId)}`);
    remoteDoc = data || current;
  }
  const remoteContent = remoteDoc?.deleted ? '' : decodeBody(remoteDoc || {});
  const localPath = item.payload?.devicePath || doc?.localPath || '';
  const decision = this.getRemoteConflictDecision(item.fileId, remoteContent, localPath);
```

这里的处理顺序非常值得注意：

1. 先拿服务端返回的 `current`
2. 如果正文不完整，再补拉全文
3. 再与本地实时编辑状态对比
4. 如果不是真冲突，就自动吸收远端
5. 如果是真冲突，写入 `conflicts`

### 10.3 冲突对象是如何建立的

统一结构由 `buildConflict()` 生成：

代码位置：[syncEngine.js:L602-L613](../src/services/syncEngine.js#L602-L613)
```js
buildConflict(fileId, remoteDoc, remoteContent, localContentOverride) {
  const doc = useSyncStore.getState().getDoc(fileId);
  return {
    fileId,
    path: doc?.localPath || fileId,
    name: doc?.name || remoteDoc?.fileName || fileId,
    localContent: typeof localContentOverride === 'string'
      ? localContentOverride
      : getLocalConflictContent(doc?.localPath, fileId),
    remoteContent,
    remoteDoc,
  };
}
```

因此，冲突在这个系统里不是一个布尔值，而是一个完整对象。

### 10.4 用户如何解决冲突

用户决策最终落在 `resolveConflict()`：

```js
代码位置：[syncEngine.js:L1084-L1100](../src/services/syncEngine.js#L1084-L1100)
async resolveConflict(fileId, resolution) {
  const conflict = useSyncStore.getState().listConflicts().find((item) => item.fileId === fileId);
  if (!conflict) return;
  useSyncStore.getState().resolveConflict(fileId);
  if (resolution === 'remote') {
    await this.applyRemoteDoc(conflict.remoteDoc, { force: true });
  } else if (resolution === 'local') {
    const doc = useSyncStore.getState().getDoc(fileId);
    const localPath = doc?.localPath || '';
    const content = conflict.localContent || '';
    if (localPath) {
      await this.queueLocalUpsert(localPath, content, doc?.encoding || 'UTF-8', {
        name: doc?.name || basename(localPath),
        lineEnding: doc?.lineEnding || 'LF',
        source: 'conflict',
      });
    }
  }
}
```

这段代码非常关键，因为它说明：

- 选远端不是改个状态，而是重新应用远端文档
- 选本地也不是直接写服务端，而是重新入队成一次新的合法 upsert

这让冲突解决与正常同步流程保持一致。

## 13. 真正吃透这一块时，你要重点关注哪些“设计决定”

最后给你一个“抓重点”的阅读视角。  
如果只是看语句，很容易迷失；但如果抓住下面这些设计决定，整个同步模块会清晰很多。

### 13.1 为什么需要 `fileId`

因为本地路径不稳定：

- 可以重命名
- 可以移动
- 不同设备路径不同

所以项目用 `fileId` 作为云端稳定身份，用本地路径做设备侧映射。

### 13.2 为什么需要 `lastKnownServerRev`

因为前端要知道“下一次写入是基于哪个远端版本产生的”，否则就没法做乐观并发控制。

### 13.3 为什么需要 `mutationId`

因为同步系统天生会遇到：

- 自动重试
- 用户重复触发
- 网络超时

没有幂等 ID，就无法区分“新操作”和“旧操作重发”。

### 13.4 为什么需要 external 文档模式

因为云文档不一定在每台设备都有真实本地路径。  
如果强行要求所有设备都先落地到文件系统，很多同步场景会变得非常笨重。

### 13.5 为什么不做自动三方合并

因为当前架构更偏向“保护本地编辑意图”，而不是“多人实时协作编辑器”。  
所以它更愿意显式生成冲突，让用户决定，而不是冒险静默 merge。

## 15. 结论

同步模块的本质，不是“调几个接口”，而是一个完整的跨端协作系统：

- 前端用 `syncEngine` 统一调度本地保存、入队、推送、拉取、冲突、配置同步
- `useSyncStore` 负责维护文档镜像、mutation 队列和冲突状态
- `apiClient` 负责把同步语义转换成带鉴权的 HTTP 调用
- 后端 `SyncController` 提供 `/sync/*` 接口面
- 后端 `SyncService` 负责版本控制、幂等处理、路径绑定、墓碑删除和配置读写
- `SyncDocument` 与 `SyncConfig` schema 决定了整个系统的持久化结构

如果你把这几个文件真正串起来读完，基本就算把当前项目的同步模块吃透了。
