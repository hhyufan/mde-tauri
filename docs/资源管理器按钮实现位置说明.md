# 资源管理器按钮实现位置说明

这篇文档专门回答这些功能分别写在哪一层：

- 上一级
- 前进
- 后退
- 排序
- 刷新
- 保存
- 打开资源管理器

文中尽量按“按钮入口 -> 前端逻辑 -> 原生/底层实现”的顺序展开，方便你直接顺着代码读。

## 1. 这些按钮大体分布在哪

资源管理器相关按钮并不都在同一个文件里。

- 资源管理器顶部导航区的“前进 / 后退 / 上一级 / 刷新 / 打开资源管理器 / 关闭目录”主要写在 [../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L324-L384)。
- 资源管理器标题栏上的“排序 / 新建文件 / 保存 / 打开文件夹”主要写在 [../src/components/layout/sidebar/Sidebar.jsx](../src/components/layout/sidebar/Sidebar.jsx#L102-L143)。
- 真正执行目录加载、保存文件、打开系统文件管理器的公共动作，集中收口在 [../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L153-L1065)。
- 与 Tauri 原生桥接相关的统一 API 在 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L100-L367)。
- 桌面端真正落到 Rust command 的保存、打开系统资源管理器逻辑在 [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L195-L259) 和 [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L890-L949)。

所以这块的阅读顺序建议是：

1. 先看 `Sidebar.jsx` 和 `FileTree.jsx`，搞清楚按钮是谁渲染出来的。
2. 再看 `useFileManager.js`，搞清楚按钮点下去之后到底触发了什么。
3. 最后看 `tauriApi.js` 和 `src-tauri/src/lib.rs`，搞清楚桌面端最终怎么落到系统能力。

## 2. 上一级按钮写在哪

“上一级”按钮本身定义在 `FileTree` 顶部导航栏里，按钮点击时会触发 `handleGoUp`：

源码入口：
[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L346-L353)

```jsx
{navBtn(
  t('sidebar.explorer.up'),
  'file-tree__nav-btn',
  handleGoUp,
  currentDirIsSaf,
  <svg viewBox="0 0 24 24" fill="none" stroke="currentColor" strokeWidth="2">
    <line x1="12" y1="19" x2="12" y2="5" /><polyline points="5 12 12 5 19 12" />
  </svg>,
)}
```

`handleGoUp` 本体也在同一个文件里：
[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L201-L211)

```js
const handleGoUp = useCallback(() => {
  if (!currentDir) return;
  if (currentDirIsSaf) return;
  const parentDir = getParentDir(currentDir);
  if (parentDir) loadDirectory(parentDir);
}, [currentDir, currentDirIsSaf, loadDirectory]);
```

这里的关键不是直接自己刷新文件列表，而是：

- 先用 `getParentDir()` 计算父目录路径
- 再调用 `loadDirectory(parentDir)` 切换到父目录

父目录路径的计算逻辑在同文件的 [../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L42-L61)，这里专门兼容了 Windows 盘符路径，比如 `C:\` 这种根目录边界。

真正加载目录的动作不在 `FileTree.jsx` 内部，而在 `useFileManager`：
[../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L356-L370)

```js
const loadDirectory = useCallback(async (dirPath) => {
  try {
    const contents = await getDirectoryContents(dirPath);
    setFiles(sortDirectoryContents(contents));
    setCurrentDir(dirPath);
  } catch (err) {
    notify('error', t('notification.error'), String(err));
  }
}, [sortDirectoryContents]);
```

所以“上一级”的完整链路是：

`FileTree.jsx` 的按钮 -> `handleGoUp()` -> `getParentDir()` -> `useFileManager.loadDirectory()`

## 3. 前进和后退按钮写在哪

前进、后退按钮也都在 `FileTree.jsx` 顶部导航栏：

- 后退按钮：[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L328-L336)
- 前进按钮：[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L337-L345)

后退逻辑：
[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L176-L187)

```js
const handleGoBack = useCallback(async () => {
  if (!canGoBack) return;
  const targetDir = dirHistory[historyIndex - 1];
  useFileStore.getState().goBack();
  await loadFilesOnly(targetDir);
}, [canGoBack, dirHistory, historyIndex, loadFilesOnly]);
```

前进逻辑：
[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L188-L199)

```js
const handleGoForward = useCallback(async () => {
  if (!canGoForward) return;
  const targetDir = dirHistory[historyIndex + 1];
  useFileStore.getState().goForward();
  await loadFilesOnly(targetDir);
}, [canGoForward, dirHistory, historyIndex, loadFilesOnly]);
```

这里有一个很重要的设计点：它们没有直接调用 `loadDirectory()`，而是调用了 `loadFilesOnly()`。

原因是：

- `loadDirectory()` 会调用 `setCurrentDir(dirPath)`，而 `setCurrentDir()` 会把目录推进历史栈
- 前进/后退本身就是在消费历史栈，如果这里再推进一次，就会把历史记录污染掉

这个行为在 `useFileStore` 里能直接看出来。

目录历史状态定义在：
[../src/store/useFileStore.js](../src/store/useFileStore.js#L54-L76)

```js
currentDir: '',
dirHistory: [],
historyIndex: -1,
files: [],
recentFiles: [],
bookmarkedPaths: [],
sortBy: 'name',
sortOrder: 'asc',
```

后退、前进状态推进逻辑在：

- [../src/store/useFileStore.js](../src/store/useFileStore.js#L78-L97)

```js
goBack: () =>
  set((state) => {
    if (state.historyIndex <= 0) return state;
    const newIndex = state.historyIndex - 1;
    return {
      historyIndex: newIndex,
      currentDir: state.dirHistory[newIndex],
    };
  }),

goForward: () =>
  set((state) => {
    if (state.historyIndex >= state.dirHistory.length - 1) return state;
    const newIndex = state.historyIndex + 1;
    return {
      historyIndex: newIndex,
      currentDir: state.dirHistory[newIndex],
    };
  }),
```

而 `loadFilesOnly()` 在 `useFileManager.js` 里的位置是：
[../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L372-L391)

```js
const loadFilesOnly = useCallback(async (dirPath) => {
  try {
    const contents = await getDirectoryContents(dirPath);
    setFiles(sortDirectoryContents(contents));
  } catch (err) {
    notify('error', t('notification.error'), String(err));
  }
}, [sortDirectoryContents]);
```

所以“前进 / 后退”的完整链路是：

`FileTree.jsx` 按钮 -> `handleGoBack()/handleGoForward()` -> `useFileStore.goBack()/goForward()` 改历史指针 -> `useFileManager.loadFilesOnly()` 刷文件列表

## 4. 排序写在哪

排序按钮入口不在 `FileTree.jsx`，而在侧边栏头部工具条 `ExplorerToolbar`。

排序下拉菜单和排序项写在：
[../src/components/layout/sidebar/Sidebar.jsx](../src/components/layout/sidebar/Sidebar.jsx#L50-L100)

这里定义了：

- 排序方向：升序 / 降序
- 排序字段：名称 / 时间 / 大小

对应按钮挂载在：
[../src/components/layout/sidebar/Sidebar.jsx](../src/components/layout/sidebar/Sidebar.jsx#L103-L143)

```jsx
<Dropdown
  open={sortOpen}
  onOpenChange={setSortOpen}
  trigger={['click']}
  placement="bottomRight"
  arrow={false}
  popupRender={renderSortDropdown({ sortBy, sortOrder, setSortBy, setSortOrder, t, close: closeSort })}
>
  <span>
    <ToolbarButton title={t('sidebar.explorer.sort')}>
      <svg viewBox="0 0 24 24" fill="none" stroke="currentColor" strokeWidth="2"><path d="M3 6h18M3 12h12M3 18h6" /></svg>
    </ToolbarButton>
  </span>
</Dropdown>
```

排序状态存储在 `useFileStore`：
[../src/store/useFileStore.js](../src/store/useFileStore.js#L57-L64)

```js
sortBy: 'name',
sortOrder: 'asc',
```

修改排序状态的方法也在 `useFileStore`：
[../src/store/useFileStore.js](../src/store/useFileStore.js#L247-L249)

```js
setSortBy: (sortBy) => set({ sortBy }),
setSortOrder: (order) => set({ sortOrder: order }),
```

前端真正执行排序有两处：

第一处是 `FileTree.jsx` 渲染当前文件列表时再按状态计算一次：
[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L277-L291)

```js
const sortedFiles = useMemo(() => {
  const direction = sortOrder === 'desc' ? -1 : 1;
  return [...files].sort((a, b) => {
    if (a.is_dir && !b.is_dir) return -1;
    if (!a.is_dir && b.is_dir) return 1;
    let cmp = 0;
    if (sortBy === 'size') cmp = (a.size || 0) - (b.size || 0);
    else if (sortBy === 'time') cmp = (a.modified || 0) - (b.modified || 0);
    else cmp = a.name.localeCompare(b.name, undefined, { numeric: true, sensitivity: 'base' });
    if (cmp === 0) cmp = a.name.localeCompare(b.name, undefined, { numeric: true, sensitivity: 'base' });
    return cmp * direction;
  });
}, [files, sortBy, sortOrder]);
```

第二处是目录刚加载回来时，`useFileManager.loadDirectory()` 和 `loadFilesOnly()` 也都会调用统一排序函数：
[../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L327-L354)

```js
const sortDirectoryContents = useCallback((contents) => {
  const { sortBy, sortOrder } = useFileStore.getState();
  const direction = sortOrder === 'desc' ? -1 : 1;
  return [...contents].sort((a, b) => {
    if (a.is_dir && !b.is_dir) return -1;
    if (!a.is_dir && b.is_dir) return 1;

    let compareValue = 0;
    if (sortBy === 'size') {
      compareValue = (a.size || 0) - (b.size || 0);
    } else if (sortBy === 'time') {
      compareValue = (a.modified || 0) - (b.modified || 0);
    } else {
      compareValue = a.name.localeCompare(b.name, undefined, { numeric: true, sensitivity: 'base' });
    }

    if (compareValue === 0) {
      compareValue = a.name.localeCompare(b.name, undefined, { numeric: true, sensitivity: 'base' });
    }
    return compareValue * direction;
  });
}, []);
```

也就是说，排序不是只写在 UI 上，而是同时体现在：

- `Sidebar.jsx`：提供排序菜单入口
- `useFileStore.js`：保存排序状态
- `FileTree.jsx`：当前列表渲染时按状态排序
- `useFileManager.js`：目录加载后也按同样规则排序

刷新按钮在 `FileTree.jsx` 顶部导航栏：
[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L356-L364)

```jsx
{navBtn(
  t('sidebar.explorer.refresh'),

  </svg>,

```

可以看到刷新本质上没有单独写一个复杂函数，而是直接重新调用当前目录的 `loadDirectory(currentDir)`。

真正的刷新逻辑还是在：
[../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L356-L370)

```js
const loadDirectory = useCallback(async (dirPath) => {
  try {
    const contents = await getDirectoryContents(dirPath);
    setFiles(sortDirectoryContents(contents));
    setCurrentDir(dirPath);
  } catch (err) {
    notify('error', t('notification.error'), String(err));
  }
}, [sortDirectoryContents]);
```

这里做了三件事：

1. 调 `getDirectoryContents(dirPath)` 重新读目录
2. 调 `sortDirectoryContents(contents)` 套用当前排序规则
3. 调 `setCurrentDir(dirPath)` 更新当前目录与导航历史

## 6. 保存写在哪

“保存”按钮入口在 `Sidebar.jsx` 的 `ExplorerToolbar`：
[../src/components/layout/sidebar/Sidebar.jsx](../src/components/layout/sidebar/Sidebar.jsx#L132-L137)

```jsx
<ToolbarButton title={t('sidebar.explorer.save')} onClick={saveCurrentFile}>
  <svg viewBox="0 0 24 24" fill="none" stroke="currentColor" strokeWidth="2"><path d="M19 21H5a2 2 0 0 1-2-2V5a2 2 0 0 1 2-2h11l5 5v11a2 2 0 0 1-2 2z" /><polyline points="17 21 17 13 7 13 7 21" /><polyline points="7 3 7 8 15 8" /></svg>
</ToolbarButton>
```

这个按钮并不自己处理保存细节，而是直接调用 `useFileManager()` 暴露出来的 `saveCurrentFile`。

`saveCurrentFile` 的位置：
[../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L818-L825)

```js
const saveCurrentFile = useCallback(async () => {
  await saveTab();
}, [saveTab]);
```

真正核心是 `saveTab()`：
[../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L757-L816)

这里面分了三类情况：

- 当前标签还没有真实路径：走 `persistUntitledTab()`，也就是首次保存
- 当前标签是已有本地文件：直接调用 `saveFile(tab.path, tab.content, tab.encoding)`
- 当前标签是外部云文档：也会在首次保存路径里做“认领”与落盘

最直接的本地保存片段是：

```js
const result = await saveFile(tab.path, tab.content, tab.encoding);
if (result.success) {
  markTabSaved(tab.id);
  notify('success', t('notification.fileSaved'), tab.name);
  syncEngine.registerLocalDocument(tab.path, {
    name: tab.name,
    ext: tab.ext,
    encoding: tab.encoding,
    lineEnding: tab.lineEnding,
  });
  await syncEngine.queueLocalUpsert(tab.path, tab.content, tab.encoding, {
    name: tab.name,
    lineEnding: tab.lineEnding,
    source: 'manual-save',
  });
}
```

再往下一层，前端统一保存 API 在：
[../src/utils/tauriApi.js](../src/utils/tauriApi.js#L139-L163)

```js
export async function saveFile(filePath, content, encoding) {
  if (isSafUri(filePath)) {
    ...
    return {
      success: true,
      message: 'File saved successfully',
      file_path: filePath,
      file_name: safDisplayName(filePath),
      encoding: encoding || 'UTF-8',
      line_ending: detectLineEnding(content),
      size: result?.size,
    };
  }
  return invoke('save_file', { filePath, content, encoding });
}
```

桌面端最终会走 Rust 的 `save_file` command：
[../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L195-L259)

这段原生逻辑做的事情是：

- 校验路径是否为空
- 若父目录不存在则先 `create_dir_all`
- 按指定编码编码内容
- `fs::write` 写回磁盘
- 返回统一的 `FileOperationResult`

所以“保存”的完整链路是：

`Sidebar.jsx` 保存按钮 -> `useFileManager.saveCurrentFile()` -> `saveTab()` -> `tauriApi.saveFile()` -> Rust `save_file`

## 7. 打开资源管理器写在哪

这里说的“打开资源管理器”，对应的是当前目录右上角导航里的“在资源管理器中打开”。

按钮入口在：
[../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L365-L373)

```jsx
{navBtn(
  t('sidebar.explorer.openInExplorer'),
  'file-tree__nav-btn',
  () => openInExplorer(currentDir),
  false,
  <svg viewBox="0 0 24 24" fill="none" stroke="currentColor" strokeWidth="2">
    <path d="M22 19a2 2 0 0 1-2 2H4a2 2 0 0 1-2-2V5a2 2 0 0 1 2-2h5l2 3h9a2 2 0 0 1 2 2z" />
  </svg>,
)}
```

真正的前端动作在 `useFileManager.js`：
[../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L899-L926)

```js
const openInExplorer = useCallback(async (path) => {
  if (isAndroid) {
    if (isSafUri(path)) {
      try {
        await showInExplorer(path);
      } catch (err) {
        notify('error', t('notification.error'), String(err));
      }
      return;
    }
    notify('info', t('notification.info', 'Info'), t('sidebar.explorer.openInExplorer'));
    return;
  }

  try {
    await showInExplorer(path);
  } catch (err) {
    notify('error', t('notification.error'), String(err));
  }
}, [isAndroid, notify, t]);
```

可以看到这里已经把平台差异处理掉了：

- Android + SAF URI：调用 Android 文件管理器
- Android 私有目录：不给系统打开，只提示
- 桌面端：直接走统一 `showInExplorer(path)`

统一桥接 API 在：
[../src/utils/tauriApi.js](../src/utils/tauriApi.js#L321-L333)

```js
export async function showInExplorer(path) {
  if (isSafUri(path)) {
    const ok = await safOpenInFileManager(path);
    if (!ok) {
      throw new Error('Failed to open in system file manager');
    }
    return 'Opened in file manager';
  }
  return invoke('show_in_explorer', { path });
}
```

桌面端最终进入 Rust：
[../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L890-L949)

这段原生逻辑会按平台分流：

- Windows：调用 `explorer`
- macOS：调用 `open`
- Linux：调用 `xdg-open`

其中 Windows 分支还区分“文件”和“目录”：

- 文件：`explorer /select, path`
- 目录：`explorer path`

所以“打开资源管理器”的完整链路是：

`FileTree.jsx` 按钮 -> `useFileManager.openInExplorer()` -> `tauriApi.showInExplorer()` -> Rust `show_in_explorer`

## 9. 最推荐的阅读顺序

如果你想最快吃透这一组按钮，我建议按下面顺序看：

1. 先看 [../src/components/layout/sidebar/Sidebar.jsx](../src/components/layout/sidebar/Sidebar.jsx#L102-L143)，搞清楚资源管理器标题栏有哪些按钮。
2. 再看 [../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L172-L384)，搞清楚顶部导航按钮和文件树交互。
3. 再看 [../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L327-L391) 和 [../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L757-L926)，搞清楚目录加载、保存、打开系统文件管理器这些统一动作。
4. 最后看 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L139-L163) 与 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L321-L333)，以及对应的 [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L195-L259) 和 [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L890-L949)，搞清楚桌面端最终怎么落到底层。

这样看下来，你会很容易分清楚：

- 哪些只是按钮入口
- 哪些是前端状态流转
- 哪些是平台桥接
- 哪些是真正和系统文件能力打交道的原生实现
