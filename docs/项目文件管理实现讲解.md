# 我们项目如何进行文件管理的

## 1. 文档范围

这篇文档专门讲当前项目的“文件管理”实现，不展开云同步。
这里的文件管理，至少覆盖你图里提到的三大块：

- 打开方式
  - 文件关联
  - 拖拽上传
  - 文件对话框
- 文件搜索
  - 按内容搜索
  - 按文件搜索
- 侧边栏导航
  - 资源管理器
  - 文件大纲
  - 最近文件

另外还补充几个真正会影响文件管理体验的部分：

- 保存、另存为、未命名文档落盘
- 文件重命名、删除、移动
- 目录历史与面包屑导航
- 桌面端与 Android 端的跨平台文件访问桥接
- 文件监听与未保存保护

如果只看一句话，总体结构是：

`UI 入口 -> useFileManager -> store -> tauriApi/androidSaf -> Rust 文件系统命令`

## 2. 整体架构：文件管理不是单个组件，而是一条完整链路

当前项目的文件管理主编排集中在 [../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L153-L1072)。
它不是一个简单的 Hook，而是把“打开、保存、另存为、打开目录、拖拽、资源管理器新建、显示到系统资源管理器”等能力统一收口。

从调用关系上看，可以分成 4 层：

1. UI 入口层
   - [../src/App.jsx](../src/App.jsx#L147-L175)
   - [../src/components/layout/sidebar/Sidebar.jsx](../src/components/layout/sidebar/Sidebar.jsx#L379-L463)
   - [../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L69-L540)
   - [../src/components/overlays/SearchModal.jsx](../src/components/overlays/SearchModal.jsx#L29-L257)
   - [../src/components/layout/sidebar/outline/OutlineView.jsx](../src/components/layout/sidebar/outline/OutlineView.jsx#L91-L228)
2. 文件管理编排层
   - [../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L153-L1072)
3. 前端状态层
   - [../src/store/useFileStore.js](../src/store/useFileStore.js#L54-L263)
   - [../src/store/useEditorStore.js](../src/store/useEditorStore.js#L41-L478)
4. 平台桥接与原生执行层
   - [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L103-L367)
   - [../src/utils/androidSaf.js](../src/utils/androidSaf.js#L53-L169)
   - [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L947-L1044)

这条链路意味着：页面按钮和侧边栏并不直接操作文件系统，而是统一交给 `useFileManager` 协调。

## 3. 打开方式是怎么实现的

### 3.1 文件对话框打开

桌面端常规打开入口最终会落到 `useFileManager` 里的选择文件逻辑，然后再调用 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L103-L126) 暴露的读取能力。

文件读取在桥接层表现为：

代码位置：[tauriApi.js:L103-L126](../src/utils/tauriApi.js#L103-L126)
```js
export async function readFileContent(path) {
  if (isSafUri(path)) {
    return safReadFileText(path);
  }
  return invoke('read_file_content', { path });
}
```

也就是说，上层永远只说“读取这个路径”，至于是桌面路径还是 Android `content://`，由桥接层自己分流。

### 3.2 拖拽上传打开

拖拽不是一个独立文件系统模块，而是另一种“把路径送进打开链路”的入口。
对应的 UI 入口通常在 [../src/App.jsx](../src/App.jsx#L417-L478) 或编辑区外层容器里处理 `drop` 事件，然后交给 `useFileManager` 统一打开。

这类设计的好处是：

- 拖拽和文件对话框共用一套打开逻辑
- 打开后仍然统一进入 tab、最近文件、工作目录定位等后续流程

### 3.3 文件关联打开

桌面端文件关联本质上不是前端自己轮询，而是启动参数传入路径，再由 Tauri 启动链转交前端。
相关原生入口在 [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L946-L971) 里可以看到 `get_cli_args` 之类的 command 注册：

代码位置：[lib.rs:L1020-L1041](../src-tauri/src/lib.rs#L1020-L1041)
```rust
builder.invoke_handler(tauri::generate_handler![
    read_file_content,
    save_file,
    get_directory_contents,
    show_in_explorer,
    get_cli_args,
])
```

前端拿到 CLI 参数后，再走与普通打开相同的文件读取与建 tab 流程。

### 3.4 最近文件打开

最近文件不是临时 UI 状态，而是专门有 store 持久化。
相关状态一般集中在 [../src/store/useFileStore.js](../src/store/useFileStore.js#L122-L249) 或附近 store 中，打开文件后会把路径与展示名写入最近文件列表，侧边栏“最近文件”区域再直接消费这份状态。

因此“最近文件”只是另一种打开入口，不是另一套文件读取实现。

## 4. 保存、另存为、未命名文档落盘怎么走

### 4.1 保存动作的真实分层

保存动作的业务编排在 [../src/hooks/useFileManager.js](../src/hooks/useFileManager.js#L769-L898)，真正写盘在 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L142-L163)。

桥接层的保存接口类似这样：

代码位置：[tauriApi.js:L142-L163](../src/utils/tauriApi.js#L142-L163)
```js
export async function saveFile(filePath, content, encoding) {
  if (isSafUri(filePath)) {
    const result = await safWriteFileText(filePath, content);
    return {
      success: true,
      file_path: filePath,
      file_name: safDisplayName(filePath),
      encoding: encoding || 'UTF-8',
      line_ending: detectLineEnding(content),
      size: result?.size,
    };
  }
  return invoke('save_file', { filePath, content, encoding });
}
```

也就是说，前端保存不直接关心平台，只关心：

- 当前 tab 有没有绑定路径
- 当前缓冲区内容是什么
- 保存后要不要清空脏标记

### 4.2 未命名文档为什么要走另存为

未命名文档本质上还没有稳定路径，所以不能直接“覆盖保存”。
因此 `useFileManager` 在检测到文件没有路径时，会转到“另存为”链路，让用户先选择目标路径，再调用统一保存接口。

### 4.3 自动保存和脏标记

文件编辑后的脏标记通常不在文件系统层，而是在编辑器与 store 层维护。
相关状态一般分布在：

- [../src/store/useEditorStore.js](../src/store/useEditorStore.js#L218-L238)
- [../src/store/useFileStore.js](../src/store/useFileStore.js#L122-L159)
- [../src/components/editor/MonacoEditor.jsx](../src/components/editor/MonacoEditor.jsx#L183-L233)

这样做的原因是：文件管理要知道“什么时候该保存”，但不应该把编辑器输入过程和底层写盘混在一起。

## 5. 文件重命名、删除、移动怎么做

文件重命名、删除本质上是资源管理器行为，入口常在 [../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L463-L537)。

对应原生能力最终还是走 `tauriApi`：

代码位置：[tauriApi.js:L207-L230](../src/utils/tauriApi.js#L207-L230)
```js
export async function renameFile(oldPath, newPath) {
  if (isSafUri(oldPath) || isSafUri(newPath)) {
    return safRenameEntry(oldPath, newPath);
  }
  return invoke('rename_file', { oldPath, newPath });
}
```

删除也同理：

代码位置：[tauriApi.js:L236-L245](../src/utils/tauriApi.js#L236-L245)
```js
export async function deleteFile(path) {
  if (isSafUri(path)) {
    return safDeleteEntry(path);
  }
  return invoke('delete_file', { path });
}
```

真正复杂的地方不在删除命令本身，而在删除之后要同步处理：

- 文件树节点刷新
- 已打开 tab 是否关闭或标记失效
- 最近文件列表是否需要剔除
- 当前工作目录视图是否重载

这也是为什么这些动作不能只写在某个按钮回调里，而要通过统一文件管理编排层处理。

## 6. 资源管理器、面包屑、目录历史怎么联动

### 6.1 资源管理器

资源管理器主体通常在 [../src/components/layout/sidebar/explorer/FileTree.jsx](../src/components/layout/sidebar/explorer/FileTree.jsx#L69-L540)。
它展示的不是静态树，而是由当前工作目录、展开状态、选中状态共同驱动的文件视图。

目录内容来源于桥接层的统一接口：

代码位置：[tauriApi.js:L197-L202](../src/utils/tauriApi.js#L197-L202)
```js
export async function getDirectoryContents(dirPath) {
  if (isSafUri(dirPath)) {
    return safListFolder(dirPath);
  }
  return invoke('get_directory_contents', { dirPath });
}
```

所以资源管理器不需要区分桌面目录和 Android SAF 目录，只关心统一的目录条目结构。

### 6.2 面包屑导航

面包屑导航的意义不只是显示路径，而是提供“快速回到上级目录”和“当前文件定位”的能力。
这部分通常与 `useFileStore` 中的当前目录、当前文件路径、展开树节点状态联动。

### 6.3 目录历史

目录历史本质上是用户在资源管理器中的浏览轨迹缓存。
实现上往往不是浏览器式 history API，而是自己在 store 中维护一个数组和当前位置索引，便于实现“后退/前进目录”。

## 7. 文件搜索怎么做

### 7.1 按文件名搜索

文件名搜索一般基于当前工作目录树或目录索引进行过滤，入口常在 [../src/components/overlays/SearchModal.jsx](../src/components/overlays/SearchModal.jsx#L60-L127)。

如果目录树已经在内存里，文件名搜索可以直接在前端完成，不必每次都调用原生层。
这适合：

- 搜索文件名
- 快速打开文件
- 小范围工作目录检索

### 7.2 按内容搜索

内容搜索通常需要走更底层能力，特别是桌面端会交给 Rust 扫描文件系统。
相关 command 注册可以在 [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L774-L848) 看到 `search_files`：

代码位置：[lib.rs:L1020-L1041](../src-tauri/src/lib.rs#L1020-L1041)
```rust
builder.invoke_handler(tauri::generate_handler![
    search_files,
])
```

这说明全文搜索不是靠前端暴力读取所有文件内容，而是利用原生层做更合适的遍历和搜索。

### 7.3 搜索结果如何回到编辑器

搜索结果点击后，并不是只“显示一个路径”，而是会触发：

- 打开对应文件
- 激活对应 tab
- 如有命中行号，跳到对应位置
- 视情况同步资源管理器选中状态

所以搜索模块其实是文件管理链路的一个变体入口。

## 8. 文件大纲和最近文件为什么也属于文件管理

### 8.1 文件大纲

大纲视图通常在 [../src/components/layout/sidebar/outline/OutlineView.jsx](../src/components/layout/sidebar/outline/OutlineView.jsx#L91-L228)。
它依赖当前文档内容解析出的标题树，不直接操作文件系统，但它决定了用户如何在“当前文件内部”导航，因此也属于文件管理体验的一部分。

### 8.2 最近文件

最近文件虽然不碰文件树，但它是“重新进入文件”的高频入口。
只要最近文件列表里记录了路径、标题、时间戳，它就能成为一条快捷打开链路。

这也是你图里把“最近文件”放在侧边栏导航分支下的原因。

## 9. 桌面端和 Android 端为什么能共用同一套文件管理 UI

关键不是 UI 自己兼容了两套平台，而是 [../src/utils/tauriApi.js](../src/utils/tauriApi.js#L103-L367) 和 [../src/utils/androidSaf.js](../src/utils/androidSaf.js#L53-L169) 把差异吸收掉了。

例如：

- 桌面端传的通常是真实文件路径
- Android 传的通常是 `content://` URI

但到了 `useFileManager` 这一层，它们都被当成“可打开的文件标识”。

因此上层 UI 基本不需要知道：

- 这是 Windows 路径还是 Android URI
- 读写底层是 Rust 还是 Kotlin
- 当前目录是不是来自 SAF

这就是项目文件管理能跨平台复用的根本原因。

## 10. 文件监听和未保存保护怎么配合

### 10.1 文件监听

桌面端文件监听一般会交给 Rust 侧的 `notify` 等能力，对应 command 注册在 [../src-tauri/src/lib.rs](../src-tauri/src/lib.rs#L463-L561) 中可以看到：

代码位置：[lib.rs:L1020-L1041](../src-tauri/src/lib.rs#L1020-L1041)
```rust
builder.invoke_handler(tauri::generate_handler![
    start_file_watching,
    stop_file_watching,
])
```

这类监听能力用于感知：

- 外部程序修改了当前文件
- 当前目录结构变化
- 资源管理器需要刷新

### 10.2 未保存保护

未保存保护通常由编辑器状态和文件管理协同完成：

- 编辑器负责标记当前 tab 已修改
- 文件管理负责在关闭、切换、删除前判断是否要提示
- 保存成功后再统一清脏

所以它既不是纯编辑器功能，也不是纯文件系统功能，而是二者之间的协作机制。

## 12. 结论

当前项目的文件管理不是“文件树 + 打开文件”这么简单，而是一整套围绕文档生命周期构建的系统：

- 打开入口统一收口
  - 文件关联、拖拽、对话框、最近文件本质上都会汇入同一条打开链路
- 文件操作统一编排
  - 保存、另存为、重命名、删除、目录刷新不是散落实现，而是集中在 `useFileManager` 与 store 协作里
- 导航能力统一围绕文件展开
  - 资源管理器、搜索、大纲、最近文件其实都是文件定位与跳转能力
- 平台差异被压在桥接层
  - 上层 UI 和业务大体只处理统一语义，底层再由 `tauriApi`、Android SAF、Rust command 完成平台分流

所以从工程设计上看，这个项目的文件管理真正重要的不是某一个按钮怎么写，而是它把“打开、编辑、保存、搜索、导航、监听”组织成了一条一致的文件工作流。
