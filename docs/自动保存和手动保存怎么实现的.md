# 自动保存和手动保存怎么实现的

## 1. 这篇文档讲什么

这篇文档专门解释当前项目里“保存”这一块的实现方式，重点回答 4 个问题：

- 手动保存从哪里触发
- 自动保存从哪里触发
- 前端是如何决定“保存到哪里”的
- 真正的落盘是在什么地方完成的

这里不只讲 UI 现象，而是把调用链拆成“编辑器 -> 文件管理层 -> 平台桥接层 -> Rust/Android 原生层”。

---

## 2. 保存能力的整体分层

当前项目的保存实现并不是让每个编辑器组件自己直接写文件，而是统一收敛到文件管理层。

- 编辑器层负责发现“内容变了”
- `useFileManager` 负责决定这是手动保存还是自动保存、是已有路径还是未命名标签、是本地文件还是 external 文档
- `tauriApi` 负责把“保存文件”分流到桌面端 Rust command 或 Android SAF
- 桌面端真正的文件落盘入口是 Rust 的 `save_file`

相关代码入口直接分布在下面几处：

- 手动保存入口：[App.jsx:L177-L229](../src/App.jsx#L177-L229)
- 侧边栏保存按钮：[Sidebar.jsx:L103-L142](../src/components/layout/sidebar/Sidebar.jsx#L103-L142)
- 自动保存总编排：[useFileManager.js:L301-L325](../src/hooks/useFileManager.js#L301-L325)
- 保存桥接层：[tauriApi.js:L139-L163](../src/utils/tauriApi.js#L139-L163)
- Rust 落盘命令：[lib.rs:L192-L259](../src-tauri/src/lib.rs#L192-L259)

## 3. 手动保存是怎么触发的

手动保存主要有两条入口。

### 3.1 全局快捷键触发

在 [App.jsx:L177-L229](../src/App.jsx#L177-L229) 里，`handleKeyDown()` 会拦截 `Ctrl/Cmd + S`，然后调用 `saveCurrentFile()`。

代码位置：[App.jsx:L199-L202](../src/App.jsx#L199-L202)

```js
if (mod && e.key.toLowerCase() === 's') {
  e.preventDefault();
  saveCurrentFile();
}
```

这说明快捷键本身并不直接操作文件系统，它只是把动作交给文件管理层。

### 3.2 侧边栏按钮触发

另一条入口在 [Sidebar.jsx:L103-L142](../src/components/layout/sidebar/Sidebar.jsx#L103-L142) 的工具栏按钮上，点击后同样调用 `saveCurrentFile()`。

因此，无论用户是按快捷键还是点按钮，最后都会汇总到同一个保存函数里，这样可以保证保存逻辑不会分散在多个组件里。

## 4. `saveCurrentFile()` 最终怎么落到真正保存逻辑

`saveCurrentFile()` 本身只是一个轻量包装，真正做事的是 `saveTab()`。

代码位置：[useFileManager.js:L828-L835](../src/hooks/useFileManager.js#L828-L835)

```js
const saveCurrentFile = useCallback(async () => {
  const tab = useEditorStore.getState().getActiveTab();
  if (!tab) return false;
  return saveTab(tab.id);
}, [saveTab]);
```

再往下看，`saveTab()` 才是手动保存的主入口，它会根据当前标签状态分流：

- 如果是纯云端 external 文档，则不走本地落盘
- 如果是未命名文档，则先决定目标路径
- 如果已有本地路径，则直接保存到该路径

代码位置：[useFileManager.js:L769-L826](../src/hooks/useFileManager.js#L769-L826)

这个设计的关键意义是：UI 入口统一，但底层处理分支完全不同。

---

## 5. 未命名文档手动保存是怎么处理的

如果当前标签还没有真实路径，`saveTab()` 不会直接写文件，而是先走 `persistUntitledTab()`。

代码位置：[useFileManager.js:L791-L799](../src/hooks/useFileManager.js#L791-L799)

```js
if (!tab.path) {
  const result = await persistUntitledTab(tab, {
    allowDialog: true,
    source: 'manual-save',
  });
  return !!result;
}
```

`persistUntitledTab()` 的职责是：

- 判断能不能直接保存到当前资源管理器目录
- 如果不能，则弹出“另存为”路径选择
- 取得真实路径后调用 `saveFile()`
- 保存成功后再由 `finalizeSavedTab()` 更新标签状态

相关代码位置：

- 未命名标签落盘：[useFileManager.js:L546-L601](../src/hooks/useFileManager.js#L546-L601)
- 首次保存后的状态收敛：[useFileManager.js:L462-L493](../src/hooks/useFileManager.js#L462-L493)

这也是为什么“新建文件第一次保存”和“已有文件再次保存”在用户看来都叫“保存”，但代码里其实是两条链路。

## 6. 自动保存是怎么触发的

自动保存不是 `useFileManager` 主动轮询触发的，而是编辑器在内容变化后，通过一个短延迟回调通知上层。

### 6.1 Monaco 编辑器里的自动保存触发

在 [MonacoEditor.jsx:L210-L233](../src/components/editor/MonacoEditor.jsx#L210-L233) 里，每次内容变化都会：

- 先把最新文本写入 `editorBuffer`
- 再节流触发脏标记、字数统计和自动保存

代码位置：[MonacoEditor.jsx:L210-L233](../src/components/editor/MonacoEditor.jsx#L210-L233)

```js
const scheduleAutoSave = () => {
  if (autoSaveScheduled) return;
  autoSaveScheduled = true;
  setTimeout(() => {
    autoSaveScheduled = false;
    onAutoSaveRef.current?.();
  }, 300);
};

editor.onDidChangeModelContent(() => {
  if (suppressModelChangeRef.current) return;
  const tabId = currentTabIdRef.current;
  if (!tabId) return;
  const value = editor.getValue();
  setBuffer(tabId, value);
  scheduleDirtyMark();
  scheduleCharCount();
  scheduleAutoSave();
});
```

这里的重点是：编辑器只负责“告诉上层内容变了”，并不自己直接保存。

### 6.2 Milkdown 编辑器里的自动保存触发

Milkdown 也是同样思路，只是实现函数名不同。

代码位置：[MilkdownMarkdownEditor.jsx:L1373-L1399](../src/components/editor/MilkdownMarkdownEditor.jsx#L1373-L1399)

```js
const scheduleEditorSideEffects = useCallback((markdown) => {
  const tabId = currentTabIdRef.current;
  if (!tabId || readOnly) return;

  setBuffer(tabId, markdown);

  if (!dirtyTimerRef.current) {
    dirtyTimerRef.current = setTimeout(() => {
      dirtyTimerRef.current = null;
      markTabDirty(tabId, true);
    }, 200);
  }

  if (!saveTimerRef.current) {
    saveTimerRef.current = setTimeout(() => {
      saveTimerRef.current = null;
      onAutoSaveRef.current?.();
    }, 300);
  }
}, [markTabDirty, readOnly, setCharacterCount]);
```

也就是说，Monaco 和 Milkdown 虽然编辑体验不同，但自动保存的协作协议是一致的：编辑器只抛出 `onAutoSave`。

## 8. `triggerAutoSave()` 是自动保存的真正总入口

自动保存最核心的函数在 [useFileManager.js:L301-L325](../src/hooks/useFileManager.js#L301-L325)。

代码位置：[useFileManager.js:L301-L325](../src/hooks/useFileManager.js#L301-L325)

```js
const triggerAutoSave = useCallback(() => {
  const autoSave = useConfigStore.getState().autoSave;
  if (!autoSave) return;
  const tab = useEditorStore.getState().getActiveTab();
  if (!tab) return;
  const content = getBuffer(tab.id, tab.content);
  if (!tab.path && tab.externalFileId) {
    debouncedExternalSync(tab.id, tab.externalFileId, content, tab.encoding, {
      name: tab.name,
      lineEnding: tab.lineEnding,
    });
    return;
  }
  if (!tab.path) {
    const canSaveToExplorerDir = hasOpenExplorerDirectory();
    if (!canSaveToExplorerDir && dismissedAutoSavePromptRef.current.has(tab.id)) return;
    persistUntitledTab(tab, { allowDialog: !canSaveToExplorerDir, source: 'auto-save' });
    return;
  }
  debouncedAutoSave(tab.path, content, tab.encoding, {
    name: tab.name,
    ext: tab.ext,
    lineEnding: tab.lineEnding,
  });
}, [debouncedAutoSave, debouncedExternalSync]);
```

这里可以看出它会先做配置与上下文判断，再决定走哪条链路：

- `autoSave` 关闭则直接返回
- external 文档走远端同步链路
- 未命名标签走 `persistUntitledTab()`
- 已有本地路径走 `debouncedAutoSave()`

这就是整个自动保存的调度中枢。

## 9. 已有本地路径的自动保存如何真正落盘

本地已有路径的自动保存并不是立刻写文件，而是先经过一个防抖函数 `debouncedAutoSave()`。

代码位置：[useFileManager.js:L242-L267](../src/hooks/useFileManager.js#L242-L267)

```js
const debouncedAutoSave = useMemo(
  () => debounce(async (filePath, content, encoding, meta = {}) => {
    if (!filePath) return;
    try {
      const result = await saveFile(filePath, content, encoding);
      if (result.success) {
        const tab = useEditorStore.getState().getActiveTab();
        if (tab && tab.path === filePath) {
          markTabSaved(tab.id);
        }
        syncEngine.registerLocalDocument(filePath, {
          name: meta.name || filePath.split(/[\\/]/).pop() || '',
          ext: meta.ext || '',
          encoding,
          lineEnding: meta.lineEnding || 'LF',
        });
        await syncEngine.queueLocalUpsert(filePath, content, encoding, {
          name: meta.name || filePath.split(/[\\/]/).pop() || '',
          lineEnding: meta.lineEnding || 'LF',
          source: 'auto-save',
        });
      }
    } catch (_) { /* silent */ }
  }, 1000),
  []
);
```

这段逻辑很关键，它不只是保存文件，还做了两件重要后处理：

- `markTabSaved(tab.id)`：把标签从脏状态改回已保存
- `queueLocalUpsert(... source: 'auto-save')`：把这次保存纳入同步体系

所以“自动保存”在项目里并不是单纯的磁盘写入，而是“落盘 + 状态回写 + 同步登记”。

## 10. 前端真正调用哪个保存 API

无论是手动保存还是自动保存，本地文件最终都要走 `tauriApi.saveFile()`。

代码位置：[tauriApi.js:L142-L163](../src/utils/tauriApi.js#L142-L163)

```js
export async function saveFile(filePath, content, encoding) {
  if (isSafUri(filePath)) {
    try {
      const result = await safWriteFileText(filePath, content);
      return {
        success: true,
        message: 'File saved successfully',
        file_path: filePath,
        file_name: safDisplayName(filePath),
        encoding: encoding || 'UTF-8',
        line_ending: detectLineEnding(content),
        size: result?.size,
      };
    } catch (e) {
      return {
        success: false,
        message: `Failed to save file: ${e?.message || e}`,
      };
    }
  }
  return invoke('save_file', { filePath, content, encoding });
}
```

这说明桥接层会自动分流：

- 如果是 Android 的 `content://` SAF URI，则前端直接走 `safWriteFileText`
- 否则走 Tauri `invoke('save_file')`

因此，保存虽然看起来是统一 API，但底层执行路径在桌面端和 Android 端并不相同。

## 11. 桌面端真正的落盘入口在哪里

桌面端真正写文件的地方在 Rust command `save_file` 中。

代码位置：[lib.rs:L192-L259](../src-tauri/src/lib.rs#L192-L259)

这里负责完成的事情包括：

- 规范化目标路径

这意味着前端并不直接使用 Node 文件 API 或浏览器 API，而是通过 Tauri command 把保存动作下沉到 Rust 层完成。

## 12. 手动保存和自动保存的差异在哪里

虽然两者最后都会落到 `saveFile()`，但它们的上游策略并不相同。

### 12.1 触发方式不同

- 手动保存来自显式用户动作
- 自动保存来自编辑器内容变更后的节流回调

### 12.2 未命名文件的处理不同

- 手动保存默认允许主动弹出保存对话框
- 自动保存会先判断是否已有资源管理器目录可落盘，避免用户每敲几个字就被频繁打断

### 12.3 同步来源标记不同

代码里会把来源写进同步元数据：

- 手动保存常见 `source: 'manual-save'`
- 自动保存会写 `source: 'auto-save'`

这对于后续同步分析、状态展示和调试都很重要。

## 14. 结论

当前项目里的“保存”不是一个单点函数，而是一条完整链路：

- 编辑器负责发现内容变化
- `useFileManager` 负责保存策略决策
- `tauriApi` 负责跨平台桥接
- Rust `save_file` 负责桌面端真正落盘

所以真正需要掌握的不是“哪个函数叫 save”，而是理解这 4 层是怎么协作的。只要把 `saveTab()`、`triggerAutoSave()`、`tauriApi.saveFile()` 和 Rust `save_file` 这几处吃透，整套保存机制就基本明白了。
