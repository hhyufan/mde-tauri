# 全项目代码注释补全 Spec

## Why
当前项目横跨 React 前端、NestJS 服务端与 Tauri Rust 端，代码规模较大且模块分散。缺少统一的文档注释与关键功能注释会提高理解、维护与交接成本，尤其影响公共接口、核心流程和复杂逻辑的可读性。

## What Changes
- 为前端、服务端、Tauri 端现有代码补充统一风格的文档注释
- 为导出函数、类、Hook、Store、服务模块、控制器、DTO、Rust 公开函数等补充职责说明
- 为复杂分支、状态同步、平台兼容、编辑器行为、鉴权与数据流等关键逻辑补充必要的功能注释
- 避免为显而易见的简单赋值、样板代码和纯样式文件添加低价值注释
- 保持现有行为不变，不以“补注释”为名修改业务逻辑

## Impact
- Affected specs: 代码可维护性、开发者可读性、跨端协作可理解性
- Affected code: `src/`、`mde-server/src/`、`src-tauri/src/` 下的源码文件，以及必要时少量配置型源码文件

## ADDED Requirements
### Requirement: 为核心代码单元补充文档注释
系统 SHALL 为项目中的核心代码单元提供简洁且准确的文档注释，至少覆盖公开导出、跨模块调用点、业务入口与复杂职责边界。

#### Scenario: 为公开导出补充注释
- **WHEN** 开发者查看导出函数、类、Hook、Store、控制器、服务、DTO 或 Rust 公开函数
- **THEN** 可以直接从文档注释中理解其职责、输入输出或使用场景

#### Scenario: 保持注释与实现一致
- **WHEN** 为现有代码补充文档注释
- **THEN** 注释内容应准确反映当前实现，不得描述不存在的行为

### Requirement: 为复杂逻辑补充必要功能注释
系统 SHALL 在复杂、隐含约束强或跨模块协作明显的代码段附近提供必要的功能注释，帮助理解“为什么这样实现”。

#### Scenario: 为关键流程补充功能说明
- **WHEN** 代码涉及同步流程、鉴权流程、编辑器状态协调、平台差异处理或非直观算法
- **THEN** 应在对应逻辑附近提供简洁功能注释，说明约束、意图或关键步骤

#### Scenario: 避免低价值注释
- **WHEN** 代码逻辑为直接赋值、简单映射、显而易见的条件判断或纯样板结构
- **THEN** 不应添加重复代码字面含义的注释

## MODIFIED Requirements
### Requirement: 代码修改的行为边界
现有代码维护流程在执行“注释补全”类任务时，必须仅新增或调整注释内容，不得引入业务逻辑变更；若发现实现与预期不一致，应单独记录并另行处理。

## REMOVED Requirements
### Requirement: Old Feature
**Reason**: 本次变更不涉及移除既有功能或要求
**Migration**: 无
