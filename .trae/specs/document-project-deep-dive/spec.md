# 项目综合技术文档 Spec

## Why
当前项目同时包含 React 前端、Tauri 桌面壳、Rust 原生桥接与 NestJS 服务端，技术面广且上下游关系复杂。对于完全不了解项目的新读者，缺少一份从概念、架构、目录、运行方式到核心链路的综合性技术文档，会显著抬高理解、接手与排障成本。

## What Changes
- 新增一份面向陌生读者的综合性项目技术文档，系统解释项目目标、架构、目录、模块职责与关键链路
- 文档需覆盖前端 React/Tauri、服务端 NestJS、Tauri Rust 桥接、数据流、同步机制、编辑器体系、平台差异与开发流程
- 文档需以“从 0 到深入”的阅读路径组织内容，先讲整体再讲细节，避免假定读者已有上下文
- 文档需解释关键术语、运行依赖、环境准备、常见开发命令与调试入口
- 文档需明确模块之间如何协作，并给出关键流程的文字化拆解
- 文档需保持与当前代码实现一致，不得虚构不存在的模块、能力或行为

## Impact
- Affected specs: 项目可交接性、团队 onboarding 能力、跨端架构可理解性、维护与排障效率
- Affected code: 项目根目录文档体系，以及 `src/`、`src-tauri/src/`、`mde-server/src/`、关键配置文件所对应的说明内容

## ADDED Requirements
### Requirement: 提供面向零基础读者的项目全景说明
系统 SHALL 提供一份单一入口的综合技术文档，使完全不了解该项目的读者能够先建立整体认知，再逐步进入实现细节。

#### Scenario: 新读者首次阅读项目
- **WHEN** 读者第一次打开该综合文档
- **THEN** 文档应先说明项目是什么、解决什么问题、由哪些子系统组成，以及推荐的阅读顺序

#### Scenario: 读者需要快速建立架构全貌
- **WHEN** 读者希望了解前端、桌面壳、原生桥接与服务端之间的关系
- **THEN** 文档应清晰描述各层职责、调用方向与数据流边界

### Requirement: 覆盖项目关键模块与核心技术链路
系统 SHALL 在文档中覆盖项目的关键模块、目录结构、核心能力与主要运行链路，且说明必须可映射到现有实现。

#### Scenario: 读者查看模块职责
- **WHEN** 读者阅读某一层级或目录的说明
- **THEN** 文档应解释该目录的职责、主要文件类型、核心模块及它们之间的关系

#### Scenario: 读者查看关键流程
- **WHEN** 读者需要理解文件管理、编辑器渲染、同步、鉴权或平台桥接等流程
- **THEN** 文档应提供按步骤展开的流程说明，帮助读者理解“谁触发、谁处理、结果写到哪里”

### Requirement: 覆盖开发、运行与调试入口
系统 SHALL 在文档中提供新开发者完成本地认知与上手所需的必要操作信息。

#### Scenario: 新开发者准备运行项目
- **WHEN** 读者想在本地启动项目前端、桌面端或服务端
- **THEN** 文档应说明依赖前提、关键命令、主要配置点以及不同运行方式的区别

#### Scenario: 读者定位调试入口
- **WHEN** 读者需要排查某个功能问题
- **THEN** 文档应指出应优先查看的模块、状态层、桥接层或服务端入口

### Requirement: 提供术语、边界与已知约束说明
系统 SHALL 对文档中涉及的重要术语、平台边界与理解门槛较高的约束提供显式解释。

#### Scenario: 读者不理解项目术语
- **WHEN** 文档中出现如 Tauri、SAF、同步引擎、编辑器缓冲区等专有概念
- **THEN** 文档应提供简明定义，并说明该概念在本项目中的作用

#### Scenario: 读者评估能力边界
- **WHEN** 读者关心某功能由哪一端负责、哪些行为依赖平台能力
- **THEN** 文档应明确说明职责归属、平台差异与实现限制

## MODIFIED Requirements
### Requirement: 项目文档的可读性要求
现有项目文档体系在新增综合性技术文档时，必须以陌生读者为目标对象组织内容，采用“概览 -> 架构 -> 模块 -> 流程 -> 开发调试 -> 术语补充”的渐进式结构，避免仅堆砌目录或源码细节。

## REMOVED Requirements
### Requirement: Old Feature
**Reason**: 本次变更不涉及移除既有功能或要求
**Migration**: 无
